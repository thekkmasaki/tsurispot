7 Route 53（DNS）〜ドメインとトラフィック管理〜

7.1 DNSとは何か？

DNS（Domain Name System）とは、人間が覚えやすいドメイン名（例: tsurispot.com）を、コンピュータが理解できるIPアドレス（例: 54.199.123.45）に変換する仕組みです。インターネットの「電話帳」のようなものです。

DNSは「電話帳」: あなたが友人に電話をかけるとき、「田中太郎」という名前で検索して電話番号を調べますよね？ DNSも同じで、「tsurispot.com」という名前から「54.199.123.45」というIPアドレス（電話番号に相当）を調べます。もしDNSがなかったら、すべてのWebサイトにIPアドレスを直接入力しなければならず、とても不便です。

ブラウザに tsurispot.com と入力すると、裏側では以下のステップでIPアドレスが解決されます。

7.2 Route 53 の基本概念

Route 53 は AWS のマネージド DNS サービスです。名前の「53」はDNSが使うポート番号53に由来します。3つの機能を提供します。

  • ドメイン登録: 新しいドメイン（例: tsurispot.com）を購入・管理
  • DNS ルーティング: ドメイン名をIPアドレスやAWSリソースにマッピング
  • ヘルスチェック: リソースの正常性を監視し、異常時にトラフィックを切り替え

7.3 ルーティングポリシー（全7種類）

Route 53のルーティングポリシーは、DNSクエリへの応答ルールです。7種類すべてが試験範囲です。

ルーティングポリシーは「お客の案内方法」: レストランチェーンで例えると――
シンプル = 「この店に行って」（1店舗案内）
加重 = 「70%は渋谷店、30%は新宿店」（A/Bテスト）
レイテンシー = 「一番近い店へ」（最速到着）
フェイルオーバー = 「渋谷店が満席なら新宿店へ」
地理的位置 = 「東京なら渋谷店、大阪なら梅田店」
地理的近接性 = 「一番近い店（バイアス調整可）」
マルチバリュー = 「空いてる店を最大8つ案内」

問題パターン: 「複数リージョンのアプリで最も応答が速いリージョンに誘導したい」→ レイテンシーベースルーティング。「地理的位置」ではなく「レイテンシー（遅延）」がキーワード。

問題パターン: 「プライマリサイト障害時にS3の静的エラーページに切り替えたい」→ フェイルオーバールーティング + ヘルスチェック。

7.4 ヘルスチェック

Route 53のヘルスチェックは、ルーティング先のリソースが正常かどうか定期的に確認する機能です。3種類あります。

  • エンドポイントヘルスチェック: URL/IPにHTTP/HTTPS/TCPでアクセスし応答確認
  • 計算されたヘルスチェック: 複数のヘルスチェックをAND/OR条件で組み合わせ
  • CloudWatchアラーム連携: CloudWatchメトリクスのアラーム状態で判定

7.5 Route 53 Resolver

オンプレミス環境とVPCのDNS解決に使います。インバウンドエンドポイント（オンプレ→VPC）とアウトバウンドエンドポイント（VPC→オンプレ）があります。

問題パターン: 「オンプレミスからVPC内プライベートホストゾーンの名前解決」→ Route 53 Resolver インバウンドエンドポイント。

ツリスポの将来マルチリージョン展開: レイテンシーベースルーティングで東京(ap-northeast-1)と大阪(ap-northeast-3)のApp Runnerに自動振り分け。ヘルスチェック設定で、東京障害時は自動的に大阪へフェイルオーバー。2,141スポットの釣り情報が途切れることなくユーザーに届きます。

Q1. Zone Apex（example.com）をCloudFrontに向けるレコードタイプは？

答えを見る

ALIASレコード。CNAMEはZone Apex不可。ALIASはAWS独自拡張でZone Apexにも使え、無料。

Q2. 90%を現行版、10%を新版に振り分けてテストするポリシーは？

答えを見る

加重ルーティング。重み90:10でカナリアデプロイを実現。

Q3. フェイルオーバーで障害検知に必要なものは？

答えを見る

ヘルスチェック。プライマリに設定し、失敗するとセカンダリに切替。

Q4. オンプレからVPCプライベートホストゾーンの名前解決に必要なものは？

答えを見る

Route 53 Resolver インバウンドエンドポイント。オンプレ→VPCなので「インバウンド」。

第7章まとめ

  • DNSはドメイン名→IPアドレスの変換。Route 53はAWSマネージドDNS
  • ALIASはZone Apex対応・無料・AWSリソース限定。CNAMEはZone Apex不可
  • ルーティングポリシー7種: シンプル/加重/レイテンシー/フェイルオーバー/地理的位置/地理的近接性/マルチバリュー
  • レイテンシーベース=最速リージョン、フェイルオーバー=障害時切替（頻出）
  • ヘルスチェック3種: エンドポイント/計算された/CloudWatch連携
  • フェイルオーバーにはヘルスチェックが必須
  • Resolverはハイブリッド環境のDNS解決に使用

8 CloudFront（CDN）とグローバル配信

8.1 CDNとは何か？

CDN（Content Delivery Network）とは、世界中に配置されたサーバー群（エッジロケーション）にコンテンツのコピーをキャッシュし、ユーザーに最も近いサーバーから配信する仕組みです。これにより、表示速度が劇的に向上します。

CDNは「コンビニチェーン」: Amazonの倉庫（オリジン）が千葉にあるとします。北海道の人が商品を注文すると、千葉から届くには時間がかかります。でもコンビニ（エッジロケーション）が北海道にもあれば、よく売れる商品をあらかじめ在庫しておいて、すぐに渡せます。CDNもまったく同じ仕組みで、よくアクセスされるコンテンツを世界中のエッジロケーションにキャッシュしておきます。

8.2 CloudFrontの仕組み

CloudFrontはAWSのCDNサービスです。以下の3層構造で配信します。

  • エッジロケーション: 世界400箇所以上。ユーザーに最も近い場所でコンテンツを配信
  • リージョナルエッジキャッシュ: エッジロケーションとオリジンの間にある中間キャッシュ。エッジにキャッシュがない場合、ここをチェック
  • オリジン: コンテンツの元データがある場所（S3バケット、ALB、EC2、カスタムオリジン）

8.3 S3 + CloudFront + OAC

S3バケットに保存した静的ファイル（画像、CSS、JS等）をCloudFront経由で配信するのは最も一般的な構成です。このとき重要なのがOAC（Origin Access Control）です。

OACは、S3バケットへのアクセスをCloudFront経由のみに制限する機能です。以前はOAI（Origin Access Identity）が使われていましたが、OACがその後継で推奨されています。

問題パターン: 「S3バケットの静的コンテンツをCloudFront経由でのみ配信し、S3への直接アクセスを防ぎたい」→ OAC（Origin Access Control）を設定し、S3バケットポリシーでCloudFrontからのアクセスのみ許可。OAIではなくOACが最新推奨。

8.4 キャッシュ動作

CloudFrontのキャッシュは以下の要素で制御します。

  • キャッシュポリシー: 何をキャッシュキーに含めるか（ヘッダー、Cookie、クエリ文字列）
  • TTL（Time To Live）: キャッシュの有効期間。デフォルトTTL、最小TTL、最大TTL
  • Cache-Control ヘッダー: オリジンが返すHTTPヘッダーでTTLを制御
  • Invalidation（無効化）: キャッシュを強制的にクリア。パス指定（例: /images/*）

Invalidation（無効化）は有料です（月1,000パスまで無料、それ以降は有料）。頻繁な更新にはInvalidationではなく、ファイル名にバージョン番号を含める（例: style-v2.css）方法が推奨されます。

8.5 Lambda@Edge / CloudFront Functions

エッジロケーションでリクエスト/レスポンスを変換できる2つの機能があります。

8.6 HTTPS と ACM

CloudFrontで独自ドメインのHTTPSを使うには、ACM（AWS Certificate Manager）でSSL/TLS証明書を発行します。

超重要！ CloudFrontの証明書は必ずus-east-1（バージニア北部）で発行する必要があります。他のリージョンで発行した証明書はCloudFrontに適用できません。一方、ALBの証明書はALBと同じリージョンで発行します。

8.7 署名付きURL / 署名付きCookie

  • 署名付きURL: 個別ファイルへの期限付きアクセス（例: 有料コンテンツのダウンロード）
  • 署名付きCookie: 複数ファイルへの期限付きアクセス（例: 有料会員向け動画配信サイト全体）

問題パターン: 「特定の有料会員だけが動画コンテンツにアクセスできるようにしたい」→ 署名付きCookie。1ファイルなら署名付きURL、複数ファイルなら署名付きCookie。

8.8 地理的制限・Price Class・WAF

  • 地理的制限（Geo Restriction）: 国単位でアクセスを許可/拒否
  • Price Class: 配信するエッジロケーションの範囲を制限してコスト削減
    • Price Class All: 全リージョン
    • Price Class 200: ほとんどのリージョン
    • Price Class 100: 北米+欧州のみ（最安）
  • WAF連携: CloudFrontにWAF Web ACLを適用してDDoS/SQLi/XSS等を防御

ツリスポの画像配信最適化: 2,141スポット分のOGP画像・スポット写真をS3に保存し、CloudFront経由で配信。OACでS3バケットを非公開に。東京ユーザーは東京エッジから、大阪ユーザーは大阪エッジからキャッシュ済み画像を取得。TTLを86400秒（24時間）に設定し、画像更新時はファイル名にバージョンを付与（spot-123-v2.webp）。Price Classは日本ユーザー中心なので200で十分。

CloudFront FunctionsでHTTPレスポンスヘッダーにCache-ControlやX-Content-Type-Options: nosniff等のセキュリティヘッダーを追加。Lambda@Edgeで「モバイルユーザーにはWebP画像、PCにはPNG」といったUser-Agentベースの画像最適化も可能。

署名付きURL/Cookieは「入場券」: テーマパークに入るには入場券（署名付きURL/Cookie）が必要です。入場券は有効期限があり、期限切れだと入れません。署名付きURLは「1つのアトラクション専用チケット」、署名付きCookieは「フリーパス」のようなものです。

Q1. CloudFrontのSSL証明書はどのリージョンで発行する必要があるか？

答えを見る

us-east-1（バージニア北部）。CloudFrontはグローバルサービスなので、証明書もus-east-1で発行する必要がある。ALBの証明書はALBのリージョンで発行。

Q2. S3バケットへの直接アクセスを防ぎ、CloudFront経由のみにするには？

答えを見る

OAC（Origin Access Control）を設定。旧方式のOAI（Origin Access Identity）ではなくOACが推奨。S3バケットポリシーでCloudFrontのみ許可。

Q3. 有料会員だけが複数の動画ファイルにアクセスできるようにしたい場合は？

答えを見る

署名付きCookie。複数ファイルへのアクセス制御にはCookieが適切。1ファイルだけなら署名付きURL。

Q4. エッジで軽量なヘッダー操作をしたい。Lambda@EdgeとCloudFront Functionsどちらが適切？

答えを見る

CloudFront Functions。軽量（1ms以内）でコストも安い。ネットワークアクセスや重い処理が必要ならLambda@Edge。

第8章まとめ

  • CloudFrontはAWSのCDN。エッジロケーション→リージョナルエッジ→オリジンの3層構造
  • OAC（Origin Access Control）でS3への直接アクセスを防止（OAIの後継）
  • CloudFrontのSSL証明書は必ずus-east-1で発行（ACM）
  • Lambda@Edge（重い処理）vs CloudFront Functions（軽量ヘッダー操作）
  • 署名付きURL=1ファイル、署名付きCookie=複数ファイルのアクセス制御
  • Price Classで配信エッジを制限してコスト削減
  • キャッシュ無効化は有料。バージョニングファイル名が推奨
  • WAF連携でDDoS/SQLi/XSS防御が可能

9 高可用性とスケーリング

この章はSAA試験の設計問題で最も出題頻度が高いトピックです。ELB、Auto Scaling、マルチAZ、DR戦略を徹底的に理解しましょう。

9.1 高可用性（HA）と耐障害性（FT）の違い

高可用性は「予備の車」、耐障害性は「飛行機のエンジン」:
高可用性（HA: High Availability） = 車が故障したとき、すぐに予備の車に乗り換えられる。一瞬のダウンタイムはあるかもしれないが、すぐ復旧する。
耐障害性（FT: Fault Tolerance） = 飛行機のエンジンは4つあり、1つ壊れても残り3つで飛び続ける。ダウンタイムゼロ。

9.2 ELB（Elastic Load Balancing）

ELBはトラフィックを複数のターゲット（EC2、コンテナ、Lambda等）に分散するサービスです。4種類あります。

種類レイヤープロトコル特徴ユースケース
ALBL7（アプリ）HTTP/HTTPSパス・ホストベースルーティング、WebSocketWebアプリ、マイクロサービス
NLBL4（トランスポート）TCP/UDP/TLS超高性能、固定IP、低遅延ゲーム、IoT、高スループット
GLBL3（ネットワーク）IPサードパーティアプライアンス統合ファイアウォール、IDS
CLBL4/L7HTTP/TCP旧世代、非推奨レガシーのみ

ALB vs NLB の使い分け（超頻出！）:
- 「HTTPルーティング」「パスベース」「ホストベース」「WebSocket」→ ALB
- 「固定IP」「静的IP」「超低遅延」「TCP/UDP」「毎秒数百万リクエスト」→ NLB
- 「サードパーティセキュリティアプライアンス」→ GLB

ALBの重要機能

  • パスベースルーティング: /api/* → APIサーバー、/images/* → 画像サーバー
  • ホストベースルーティング: api.example.com → APIサーバー、www.example.com → Webサーバー
  • ターゲットグループ: EC2、IP、Lambda をグループ化してルーティング先に指定
  • スティッキーセッション: 同じユーザーを同じターゲットに固定（セッション維持が必要なアプリ用）
  • SNI（Server Name Indication）: 1つのALBで複数のSSL証明書をホスト
  • Connection Draining（登録解除の遅延）: ターゲットを外すとき、既存接続の処理完了を待つ

問題パターン: 「1つのALBで複数ドメインのHTTPSを処理したい」→ SNI（Server Name Indication）。ALBは複数SSL証明書をSNIで処理可能。CLBは不可。

9.3 Auto Scaling

Auto Scalingは需要に応じてEC2インスタンス数を自動調整するサービスです。

起動テンプレート vs 起動設定

  • 起動テンプレート（Launch Template）: 推奨。バージョン管理可能、複数インスタンスタイプ対応
  • 起動設定（Launch Configuration）: 旧方式、非推奨。新規利用不可

スケーリングポリシー（5種類）

種類仕組み例
ターゲット追跡指定メトリクスの目標値を維持CPU使用率50%を維持
ステップアラーム段階に応じて段階的に追加CPU 60%で+1、80%で+3
シンプルアラーム発動で固定数追加CPU 70%超で+2台
スケジュール時刻指定で台数変更毎朝9時にmin:4に増加
予測ML（機械学習）で負荷予測過去パターンから自動予測

問題パターン: 「CPU使用率を常に一定値に保ちたい」→ ターゲット追跡スケーリング。最も一般的で推奨される方式。

9.4 DR（ディザスタリカバリ）戦略

災害時の復旧戦略は4つあり、コストとRTO（復旧時間目標）のトレードオフです。

各DR戦略の詳細

  • Backup & Restore: データのバックアップだけ取り、障害時にゼロから復元。RTO:時間〜24時間。コスト最低
  • Pilot Light: DB等の最小限のコアコンポーネントだけ常時稼働。障害時に残りを起動。RTO:10分〜数時間
  • Warm Standby: 縮小版のシステムを常時稼働。障害時にスケールアップ。RTO:分単位
  • Multi-Site Active/Active: 両リージョンで完全稼働。Route 53で振り分け。RTO:ほぼゼロ。コスト最高

問題パターン: 「最小コストでDR」→ Backup & Restore。「RTOを最小に」→ Multi-Site Active/Active。「コストとRTOのバランス」→ Warm Standby。

ツリスポのApp RunnerはAWSが自動でマルチAZ + Auto Scalingを管理。もしEC2で構築するなら: ALB(2AZ) + Auto Scaling Group(min:2, desired:2, max:10) + ターゲット追跡(CPU 60%)。釣りシーズン（春4-6月、秋9-11月）はスケジュールスケーリングでmin:4に増やす。DR戦略はWarm Standby（大阪リージョンに縮小版を待機）。

ツリスポの月間50万PV目標達成時の構成案: ALBで/api/*をAPIサーバー群へ、/*をフロントサーバー群へパスベースルーティング。NLBは不要（HTTPトラフィックのみなのでALBで十分）。スティッキーセッションは不要（Next.jsはステートレスなため）。

Auto Scalingは「回転寿司のレーン」: お客が少ないときはレーンに少しだけ寿司を流し（min:2台）、混んできたらどんどん追加し（max:10台）、空いたら減らす。ターゲット追跡は「常にレーン上の寿司密度を一定に保つ」、スケジュールは「ランチタイムは事前に増やしておく」。

Q1. HTTPベースのWebアプリでパスベースルーティングが必要。ALBとNLBどちらを使うか？

答えを見る

ALB。パスベースルーティングはL7機能でALBのみ対応。NLBはL4（TCP/UDP）。

Q2. CPU使用率を常に60%に維持するスケーリングポリシーは？

答えを見る

ターゲット追跡スケーリング。目標メトリクス値を指定すると、Auto Scalingが自動調整。

Q3. 最小コストのDR戦略は？

答えを見る

Backup & Restore。バックアップだけ取得し、障害時にゼロから復元。RTOは最長だがコスト最低。

Q4. 1つのALBで複数ドメインのHTTPSをホストするために必要な機能は？

答えを見る

SNI（Server Name Indication）。ALBは複数SSL証明書をSNIで処理可能。

Q5. 固定IPが必要なロードバランサーはどれ？

答えを見る

NLB（Network Load Balancer）。NLBはAZごとに固定IP（Elastic IP割当可能）。ALBは固定IPを持てない。

第9章まとめ

  • HA=短時間で復旧、FT=ダウンタイムゼロで継続
  • ALB=L7(HTTP)、NLB=L4(TCP/UDP固定IP)、GLB=L3(アプライアンス)
  • ALBはパス/ホストベースルーティング、SNI対応。NLBは固定IP・超高性能
  • Auto Scalingの起動テンプレートが推奨（起動設定は非推奨）
  • スケーリング5種: ターゲット追跡(推奨)/ステップ/シンプル/スケジュール/予測
  • DR戦略4種: Backup&Restore→Pilot Light→Warm Standby→Multi-Site(コスト順)
  • マルチAZ（最低2AZ）が高可用性の基本

10 サーバーレスアーキテクチャ

10.1 サーバーレスとは

サーバーレスとは「サーバーが無い」という意味ではなく、サーバーの管理が不要という意味です。AWSがインフラを完全に管理し、開発者はコードだけに集中できます。3つの特徴があります。

  • サーバー管理不要: OS、パッチ適用、スケーリングすべてAWSが担当
  • イベント駆動: リクエストやファイルアップロード等のイベントに応じて実行
  • 従量課金: 使った分だけ支払い。アイドル時はゼロコスト

サーバーレスは「タクシー」、EC2は「マイカー」: マイカー(EC2)は駐車場代・保険・メンテが必要で、使わなくてもコストがかかる。タクシー(Lambda)は乗った分だけ払えばよく、車の整備は運転手（AWS）がやってくれる。短距離移動が多いならタクシーがお得、毎日長距離通勤ならマイカーがお得。

10.2 Lambda

AWS Lambdaはサーバーレスの中核サービスです。コードをアップロードするだけで、イベントに応じて自動実行されます。

Lambda の基本スペック

項目値
対応ランタイムNode.js, Python, Java, Go, .NET, Ruby, カスタムランタイム
メモリ128MB 〜 10,240MB（10GB）
タイムアウト最大15分（900秒）
同時実行数デフォルト1,000（引き上げ可能）
デプロイパッケージ最大250MB（解凍後）
一時ストレージ(/tmp)最大10GB
環境変数KMS暗号化対応

Lambda のタイムアウトは最大15分です。15分を超える処理にはLambdaは不向きです。長時間処理にはECS/Fargate、Step Functions、またはEC2を検討してください。

コールドスタート

コールドスタートとは、Lambdaが初めて実行される際（またはしばらく使われなかった後）に実行環境を準備する時間です。数百ms〜数秒かかります。

  • 対策1: プロビジョニング済み同時実行 = 指定数の実行環境を常時ウォーム状態に維持（コストは増加）
  • 対策2: Lambda Layers = 共通ライブラリを別管理し、デプロイサイズを削減
  • 対策3: VPC外に配置 = VPC内Lambdaは ENI作成 でさらに遅くなる（改善済みだが注意）

10.3 API Gateway

API GatewayはAPIのフロントドアです。RESTful APIやWebSocket APIを作成・管理します。

項目REST APIHTTP API
料金高い安い（REST APIの約70%OFF）
機能フル機能（キャッシュ、WAF、使用量プラン等）コア機能のみ
認証IAM, Cognito, Lambda AuthorizerIAM, Cognito, JWT
推奨高度な機能が必要な場合シンプルなAPI（推奨）

問題パターン: 「コストを抑えてシンプルなREST APIを構築したい」→ HTTP API。REST APIより安く高速。ただしAPIキャッシュやWAF統合が必要ならREST API。

10.4 Cognito（認証サービス）

Cognitoはユーザー認証と認可を提供するサービスです。2つのコンポーネントがあります。

  • User Pool（ユーザープール）: サインアップ/サインイン機能。MFA、ソーシャルログイン（Google/Facebook/Apple）対応。JWTトークンを発行。
  • Identity Pool（IDプール）: AWSリソースへの一時的なアクセス権を付与。User Poolで認証されたユーザーやゲストユーザーに、S3やDynamoDBへの直接アクセス権限を提供。

10.5 Step Functions / S3イベント / AppSync

  • Step Functions: 複数のLambda関数をワークフローとしてオーケストレーション（順番制御）。Standard（長時間、1年まで）とExpress（短時間、5分まで、高スループット）の2種類
  • S3イベント + Lambda: S3にファイルがアップロードされたらLambdaを自動実行（例: 画像リサイズ、サムネイル生成）
  • AppSync: マネージドGraphQL API。リアルタイムデータ同期とオフラインアクセス対応

問題パターン: 「複数のLambda関数を順番に実行し、エラー時にリトライやフォールバック処理をしたい」→ Step Functions。Lambda同士をコードで呼び出すのではなく、Step Functionsでワークフロー管理。

問題パターン: 「S3に画像がアップロードされたら自動でサムネイルを生成したい」→ S3イベント通知 + Lambda。S3のPutObjectイベントをトリガーにLambdaを起動。

ツリスポの釣果投稿機能をサーバーレス設計: ユーザー認証=Cognito User Pool（Google/Appleログイン対応）、写真アップロード=S3 + Lambda（リサイズ処理）、投稿データ保存=API Gateway → Lambda → DynamoDB、リアルタイム通知=DynamoDB Streams → Lambda → SNS（プッシュ通知）。全てサーバーレスなので、投稿がない深夜はコストゼロ。

釣果投稿ワークフローをStep Functionsで管理: (1)画像をS3にアップロード → (2)Lambda で画像リサイズ+不適切画像チェック → (3)投稿データをDynamoDBに保存 → (4)SNSでフォロワーに通知。途中で画像チェックがNGなら投稿拒否に分岐。

Q1. Lambdaの最大実行時間は？

答えを見る

15分（900秒）。それ以上かかる処理にはFargate/EC2/Step Functionsを検討。

Q2. User PoolとIdentity Poolの違いは？

答えを見る

User Pool=認証（サインイン/サインアップ、JWTトークン発行）。Identity Pool=認可（AWS一時認証情報の付与、S3/DynamoDB等への直接アクセス）。

Q3. S3に画像アップロード時に自動でリサイズするには？

答えを見る

S3イベント通知 + Lambda。PutObjectイベントでLambdaをトリガー。

Q4. 複数Lambda関数を順次実行しエラーハンドリングするサービスは？

答えを見る

Step Functions。ワークフローをJSON/YAMLで定義し、分岐・リトライ・並列実行を管理。

第10章まとめ

  • サーバーレス = サーバー管理不要 + イベント駆動 + 従量課金
  • Lambda: 最大15分、メモリ最大10GB、同時実行デフォルト1,000
  • コールドスタート対策: プロビジョニング済み同時実行
  • API Gateway: HTTP API（安い・推奨） vs REST API（フル機能）
  • Cognito User Pool=認証（JWT）、Identity Pool=認可（AWS一時認証情報）
  • Step Functions=ワークフローオーケストレーション
  • S3イベント+Lambda=ファイルアップロード時の自動処理

11 コンテナサービス

11.1 コンテナとは

コンテナとは、アプリケーションとその実行に必要なすべて（コード、ライブラリ、設定ファイル等）を1つのパッケージにまとめた軽量な仮想化技術です。Dockerが最も有名なコンテナ技術です。

コンテナは「引越し用コンテナ」: 引越しのとき、荷物をコンテナに詰めれば、どのトラックに載せても中身は同じ。開発環境で動いたアプリをコンテナに詰めれば、本番環境でもまったく同じように動く。「自分のPCでは動くのに...」という問題がなくなります。

Docker基本用語

用語意味例え
イメージコンテナの設計図（読み取り専用）レシピ本
コンテナイメージから起動した実行インスタンスレシピ通りに作った料理
Dockerfileイメージの作り方を書いたファイルレシピの手順書
レジストリイメージを保存・配布する場所レシピ本の図書館（ECR）

VM vs コンテナ vs サーバーレス

項目VM（EC2）コンテナ（ECS/EKS）サーバーレス（Lambda）
起動速度分単位秒単位ミリ秒単位
管理負担高い(OS管理)中程度低い（コードのみ）
柔軟性最高高い制約あり(15分等)
料金常時課金常時/タスク単位実行時間のみ
適合フルカスタムマイクロサービスイベント駆動処理

11.2 ECR（Elastic Container Registry）

ECRはDockerイメージの保管庫（レジストリ）です。Docker Hubのようなものですが、AWS内で完結します。

  • プライベートリポジトリ: デフォルト。IAMで認証
  • パブリックリポジトリ: ECR Public Gallery で公開可能
  • ライフサイクルポリシー: 古いイメージを自動削除してストレージコスト削減
  • 脆弱性スキャン: プッシュ時に自動でセキュリティスキャン

ツリスポの実際の構成: GitHub Actionsで docker build → ECRにイメージをpush → App Runnerが新イメージを検知して自動デプロイ。ECRのライフサイクルポリシーで「最新10個以外のイメージは自動削除」を設定してストレージを節約。

11.3 ECS（Elastic Container Service）

ECSはAWS独自のコンテナオーケストレーションサービスです。Dockerコンテナの実行・管理・スケーリングを行います。

ECSの基本概念

  • クラスター: コンテナが実行される論理的なグループ
  • タスク定義: コンテナの設計図（使用イメージ、CPU、メモリ、ポート等を定義）
  • タスク: タスク定義から起動された実行インスタンス
  • サービス: タスクの実行数を維持する仕組み（タスクが停止したら自動復旧）

起動タイプ: EC2 vs Fargate（試験頻出！）

項目EC2起動タイプFargate起動タイプ
インスタンス管理自分で管理（OS, パッチ, スケーリング）AWS完全管理（サーバーレス）
コストRI/Spot活用で安くできるやや高い
GPUサポート対応非対応
Docker Daemonアクセス可能アクセス不可
適合コスト最適化、GPU、特殊要件運用負荷削減、一般的なアプリ

問題パターン: 「コンテナを運用したいが、サーバー管理の手間を減らしたい」→ ECS + Fargate。「コストを最小化したい（RI/Spotを活用）」→ ECS + EC2起動タイプ。

11.4 EKS（Elastic Kubernetes Service）

EKSはマネージドKubernetesサービスです。KubernetesのコントロールプレーンをAWSが管理します。

  • Kubernetes の知識・エコシステムを活用したい場合に選択
  • マルチクラウド対応（同じK8sマニフェストを他クラウドでも使える）
  • EKS Fargate: KubernetesのPodをFargate上で実行（サーバーレスK8s）
  • EKS Anywhere: オンプレミスでEKSを実行

問題パターン: 「Kubernetesを使いたい」→ EKS。「AWSだけで良い、Kubernetesの知識不要」→ ECS。試験では「Kubernetes」というキーワードがあればEKS。

11.5 App Runner

App Runnerは最もシンプルなコンテナ実行サービスです。ソースコードまたはコンテナイメージから、自動でビルド・デプロイ・スケーリング・ロードバランシング・HTTPS化を行います。

App Runner = ECS + Fargate + ALB + Auto Scaling + ACM を全部ひとまとめにしたサービス。設定項目が極めて少なく、初心者でもすぐにコンテナを公開できる。ただし細かい制御はできない。

ECS/EKS/App Runnerは「引越しサービスのランク」:
EC2 = 自分でトラック運転、荷造り、荷解き全部やる（フル自力引越し）
ECS(EC2) = トラックは自分で用意、荷造り・運搬は業者（半自力）
ECS(Fargate) = トラックも業者が手配、あなたは荷物を箱に詰めるだけ
App Runner = 「住所だけ教えてください」であとは全部やってくれる（フルサービス引越し）

ツリスポの実際の構成: GitHub Actions → ECR → App Runner。App RunnerはECS+Fargateの簡易版。現在はこれで十分だが、将来的に「サイドカーコンテナでログ収集」「バッチ処理用の別コンテナ」などが必要になったらECS+Fargateに移行。Kubernetes(EKS)は大規模マイクロサービスでない限り不要。

問題パターン: 「最小の運用負荷でコンテナアプリを公開したい」→ App Runner。「サーバーレスでコンテナを運用、ただし細かい設定も必要」→ ECS + Fargate。

Q1. Docker Hubの代わりにAWS内でDockerイメージを管理するサービスは？

答えを見る

ECR（Elastic Container Registry）。プライベート/パブリックリポジトリ、脆弱性スキャン、ライフサイクルポリシー対応。

Q2. ECSのFargate起動タイプとEC2起動タイプ、サーバー管理不要なのは？

答えを見る

Fargate。EC2起動タイプはインスタンスのOS管理が必要。Fargateはサーバーレスなので管理不要。

Q3. Kubernetesを使いたい場合のAWSサービスは？

答えを見る

EKS（Elastic Kubernetes Service）。マネージドK8sサービス。K8s不要ならECSで十分。

Q4. App Runnerの位置付けは？

答えを見る

ECS+Fargate+ALB+Auto Scaling+ACMをひとまとめにした最もシンプルなコンテナサービス。コードまたはイメージから自動デプロイ。細かい制御は不可。

第11章まとめ

  • コンテナ=アプリ+依存物をパッケージ化。Docker, イメージ, Dockerfileが基本概念
  • ECR=Dockerイメージのレジストリ。ライフサイクルポリシーと脆弱性スキャン
  • ECS=AWSのコンテナオーケストレーション。EC2(自管理)とFargate(サーバーレス)の2起動タイプ
  • EKS=マネージドKubernetes。K8sが必要な場合のみ選択
  • App Runner=最もシンプルなコンテナ実行。ビルド/デプロイ/LB/HTTPS/スケーリング全自動
  • 管理負担: EC2 > ECS(EC2) > ECS(Fargate) > App Runner
  • コスト最適化: EC2起動タイプ(RI/Spot活用)、運用負荷削減: Fargate/App Runner

12 セキュリティサービス

12.1 KMS（Key Management Service）

KMSはAWSの暗号鍵管理サービスです。データの暗号化・復号に使うマスターキーを安全に管理します。

KMSは「金庫の鍵を管理する鍵番」: あなたの大切なデータ（釣果写真、ユーザー情報）を金庫（暗号化）に入れて守ります。KMSはその金庫の鍵（暗号鍵）を安全に保管・管理する鍵番のような存在。鍵を紛失したり盗まれたりしないよう、厳重に管理してくれます。

キーの種類

種類管理者ローテーションコスト

カスタマーマネージドキーユーザーが管理自動 or 手動月額 + API呼出し
AWS所有キーAWS内部で使用AWS管理無料

エンベロープ暗号化

エンベロープ暗号化とは、データを暗号化する「データキー」を、さらにマスターキー（CMK）で暗号化する二重構造です。大量データを効率的に暗号化できます。

12.2 ACM（AWS Certificate Manager）

ACMはSSL/TLS証明書を無料で発行・管理するサービスです。

  • CloudFrontの証明書はus-east-1で発行（第8章の復習）
  • ALBの証明書はALBと同じリージョンで発行
  • 自動更新: DNS検証で発行した証明書は自動更新される
  • EC2に直接インストールすることはできない（ALB/CloudFront/API Gateway経由で使用）

問題パターン: 「EC2インスタンスに直接SSL証明書を適用したい」→ ACMは不可。EC2の前にALBを置き、ALBにACM証明書を適用するか、EC2に自前で証明書をインストール。

12.3 WAF（Web Application Firewall）

WAFはWebアプリケーションを攻撃から守るファイアウォールです。

  • Web ACL: ルールのセット。リクエストを許可/拒否/カウント
  • マネージドルール: AWS や サードパーティが提供する定番ルールセット（SQLi, XSS, Bot対策等）
  • Rate-based ルール: 同一IPから一定数以上のリクエストを自動ブロック（DDoS対策）
  • 適用先: ALB、CloudFront、API Gateway、AppSync

12.4 Shield（DDoS防御）

ツリスポのセキュリティ構成: ACMでSSL証明書（App Runnerは自動管理）。WAFでSQLインジェクション/XSS防御。環境変数（MICROCMS_API_KEY等）はSSM Parameter Store の SecureString で管理（コスト無料）。GuardDutyで不正アクセス検出。ECRイメージはInspectorで脆弱性スキャン。

将来的にユーザー投稿機能を追加する際: microCMSのAPIキーのローテーションはSecrets Managerで自動化（Lambda連携で定期的にキーを再発行）。ユーザーが投稿した写真に個人情報が写り込んでいないかMacie（S3のPII検出）で自動チェック。全体のセキュリティ状態はSecurity Hubで一元監視。

セキュリティサービスは「家の防犯対策」:
Shield = 塀（外部の大規模攻撃をブロック）
WAF = 門の番人（怪しい人を入口で止める）
GuardDuty = 防犯カメラ（異常行動を検知）
Inspector = 家の定期点検（弱い箇所を発見）
KMS = 金庫（大切なものを暗号化して保管）
Secrets Manager = 鍵の定期交換業者（パスワードを定期変更）

Q1. RDSパスワードの自動ローテーションに使うサービスは？

答えを見る

Secrets Manager。自動ローテーション機能があり、RDS/Redshift/DocumentDBと統合。Parameter Storeには自動ローテーション機能がない。

Q2. S3バケット内のデータからクレジットカード番号等のPIIを検出するサービスは？

答えを見る

Amazon Macie。ML（機械学習）を使ってS3内の機密データ（PII）を自動検出。

Q3. CloudFrontのSSL証明書を管理するサービスと、発行すべきリージョンは？

答えを見る

ACM（AWS Certificate Manager）でus-east-1に発行。CloudFrontはグローバルサービスなのでバージニア北部限定。

Q4. 全AWSアカウントに無料で自動適用されるDDoS防御サービスは？

答えを見る

Shield Standard。無料で自動適用。より高度な防御にはShield Advanced（月$3,000）。

Q5. 設定値を無料で管理したい。暗号化は不要。どのサービスが最適？

答えを見る

SSM Parameter Store（Standard）。無料。暗号化が必要なら SecureString タイプ（KMS暗号化）。ローテーション不要なら Parameter Store で十分。

第12章まとめ

  • KMS=暗号鍵管理。エンベロープ暗号化でデータキー+マスターキーの二重構造
  • ACM=SSL/TLS証明書の無料発行。CloudFrontはus-east-1必須
  • WAF=SQLi/XSS/DDoS防御。ALB/CloudFront/API Gatewayに適用
  • Shield Standard=無料DDoS防御（自動適用）。Advanced=月$3,000の高度防御
  • Secrets Manager=自動ローテーション対応（RDS統合）。Parameter Store=無料で設定値管理
  • 「ローテーション」→Secrets Manager、「コスト最小」→Parameter Store
  • GuardDuty=脅威検出、Inspector=脆弱性スキャン、Macie=PII検出
  • 多層防御: ネットワーク層+アプリ層+データ層でセキュリティを重ね掛け
