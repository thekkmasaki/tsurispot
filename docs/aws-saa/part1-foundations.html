1 AWSの全体像とグローバルインフラストラクチャ

1.1 そもそもクラウドコンピューティングって何？

まず最初に、クラウドコンピューティングとは何かを理解しましょう。難しい言葉に聞こえるかもしれませんが、実はあなたも毎日使っています。Gmailでメールを送る、Google Driveにファイルを保存する、Netflixで動画を見る。これらは全て「クラウド」です。自分のパソコンの中ではなく、インターネットの向こう側にある誰かのコンピューターがデータを保存し、処理しているのです。

AWS（Amazon Web Services）は、この「インターネットの向こう側のコンピューター」を誰でも借りられるサービスです。もともとAmazon.comが自社のECサイトを動かすために構築した大規模なインフラを、2006年に外部にも開放したのが始まりです。

レンタカーに例えると：

  • オンプレミス（自前のサーバー） = 車を購入する。初期費用が高い（数百万円）。メンテナンスも自分。使わない日も駐車場代がかかる。
  • クラウド（AWS等） = レンタカーを借りる。使いたい時だけ借りて、使った分だけ払う。メンテナンスはレンタカー会社任せ。急に大きな車が必要でもすぐ交換できる。

オンプレミスとクラウドの詳細比較

項目オンプレミス（自社サーバー）クラウド（AWS等）
初期費用数百万〜数千万円0円（従量課金）
構築時間数週間〜数ヶ月数分〜数時間

CapExとOpExの違い（試験頻出ワード）：

  • CapEx（Capital Expenditure = 設備投資）: サーバーを「購入」する費用。先に大金が出ていく。
  • OpEx（Operational Expenditure = 運用費）: サーバーを「利用」する月額費用。使った分だけ払う。

クラウド移行 = CapEx から OpEx への変換。これが試験で最も問われるクラウドのメリットです。

AWSが公式に掲げるクラウドの6大メリット

  1. 設備投資を変動費に: CapEx → OpExへの変換
  2. スケールメリットの恩恵: AWSの大量購入で単価が安い
  3. キャパシティ予測が不要: 需要に応じて自動スケール
  4. 速度と俊敏性の向上: 新サーバーを数分で起動
  5. データセンター運用費が不要: アプリ開発に集中
  6. 数分でグローバル展開: 世界中のリージョンにワンクリック

問題パターン: 「企業がクラウドに移行するメリットは？」→ CapExをOpExに変換、需要予測が不要、規模の経済が頻出キーワード。

1.3 AWSグローバルインフラストラクチャ

AWSのインフラはリージョン、アベイラビリティゾーン（AZ）、エッジロケーションの3階層です。

リージョン（Region）

世界各地に設置されたデータセンター群のまとまり。2025年時点で33リージョン。各リージョンは完全に独立しています。

  • ap-northeast-1: 東京（日本企業の標準選択、AZ 3つ）
  • ap-northeast-3: 大阪（DR用途に最適、AZ 3つ）
  • us-east-1: バージニア北部（最安、新サービス先行提供）

リージョン選択の4つの考慮点：

  1. レイテンシー: ユーザーに近いリージョン
  2. コスト: リージョンで料金が異なる（us-east-1が最安）
  3. コンプライアンス: データの国外持ち出し規制
  4. サービス可用性: 新サービスはus-east-1から

アベイラビリティゾーン（AZ）

各リージョン内に2つ以上（通常3つ）。物理的に離れた独立したデータセンター群。独自の電源・冷却・ネットワークを持ち、AZ間は高速プライベート光ファイバーで接続（1ms以下のレイテンシー）。

マンションに例えると：

  • リージョン = 都市（東京、大阪）
  • AZ = 都市内の別々のマンション（独立した電気・水道）
  • エッジロケーション = コンビニ（各地に大量にあり、人気商品をすぐ届ける）

エッジロケーション

CloudFront（CDN）やRoute 53（DNS）のキャッシュサーバー設置場所。世界400箇所以上。最寄りのエッジからコンテンツ配信することで表示速度を向上させます。

ツリスポは東京リージョン（ap-northeast-1）のApp Runnerで運用中。日本のユーザーがターゲットなので東京リージョンが最適です。将来、海外展開するならus-east-1にもデプロイし、Route 53のレイテンシーベースルーティングで自動振り分けします。

ツリスポの画像をCloudFront経由で配信すれば、エッジロケーションにキャッシュされ、北海道のユーザーにも福岡のユーザーにも高速配信。5,000+ページのサイトにCDNは必須施策です。

1.6 共有責任モデル

AWSとユーザーの責任分担を定めたモデル。ほぼ確実に出題されます。

  • AWSの責任("of" the Cloud): 物理インフラ（DC、ハードウェア、ネットワーク、ハイパーバイザー）
  • ユーザーの責任("in" the Cloud): データ、OS（EC2の場合）、アプリ、IAM、暗号化設定、SG/NACL

サービスの種類で境界線が変わる！

  • IaaS（EC2）: OS以上は全てユーザー責任
  • PaaS（RDS, App Runner）: AWSがOS管理。ユーザーはデータとアプリ
  • サーバーレス（Lambda, S3）: AWSが大部分管理。ユーザーはコード/データのみ

問題パターン1: 「EC2のOSパッチは誰の責任？」→ ユーザー（IaaSだから）

問題パターン2: 「RDSのOSパッチは？」→ AWS（マネージドサービスだから）

問題パターン3: 「DCの物理警備は？」→ AWS

ツリスポのApp RunnerはPaaS寄り。AWSがOS・パッチ・スケーリングを管理。ユーザー責任はアプリコードのセキュリティ、IAM設定、環境変数の秘密管理（microCMS APIキー等）。

# 東京リージョンのAZ一覧

2 IAM（Identity and Access Management）

IAMはAWSセキュリティの基盤であり、SAA試験で最も重要な分野の一つです。「誰が」「何に」「何をできるか」をコントロールするサービスです。IAMはグローバルサービスで、リージョンに依存しません（どこで設定しても全リージョンに適用されます）。

会社のセキュリティに例えると：

  • ルートユーザー = 会社の社長（全権限を持つが、日常業務には出てこない）
  • IAMユーザー = 社員（社員証で入館、権限は役職に応じて異なる）
  • IAMグループ = 部署（開発部、営業部。部署ごとにアクセスできる場所が違う）
  • IAMロール = 来客用バッジ（一時的に権限を貸し出す仕組み）
  • IAMポリシー = 権限規定書（「開発部は3階のサーバールームに入れる」等の規則）

2.1 ルートユーザー vs IAMユーザー

ルートユーザーはAWSアカウント作成時に自動的に作られる最高権限のアカウントです。全てのAWSサービスとリソースに無制限にアクセスできます。

ルートユーザーは日常作業に絶対使わないこと！ これはAWSのベストプラクティスで、試験でも頻繁に問われます。

ルートユーザーでしかできない操作（例外）：

  • AWSアカウントの解約
  • ルートユーザーのMFA設定
  • 請求情報の変更
  • サポートプランの変更
  • IAMユーザーの請求アクセス有効化
  • S3バケットポリシーでの全拒否からの復旧

IAMユーザーは、ルートユーザーが作成する個別のユーザーアカウントです。各IAMユーザーには必要最小限の権限だけを付与します。

セキュリティのベストプラクティス

  1. ルートユーザーにMFA（多要素認証）を設定する
  2. 日常作業用のIAMユーザーを作成する
  3. ルートユーザーのアクセスキーは作らない（作成済みなら削除する）
  4. 強力なパスワードポリシーを設定する

2.2 IAMポリシーの構造

IAMポリシーはJSON形式で記述され、「誰が何をできるか」を定義します。ポリシーの構造を理解することは試験で必須です。

{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowS3ReadOnly",
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:ListBucket"
      ],
      "Resource": [
        "arn:aws:s3:::tsurispot-images",
        "arn:aws:s3:::tsurispot-images/*"
      ],
      "Condition": {
        "IpAddress": {
          "aws:SourceIp": "203.0.113.0/24"
        }
      }
    }
  ]
}

各要素の意味：

要素説明必須？
Versionポリシー言語のバージョン。常に"2012-10-17"推奨
Statement1つ以上の許可/拒否ルールの配列必須
Sidステートメントの識別名（任意の文字列）任意
Effect"Allow" または "Deny"必須
Action許可/拒否するアクション（例: s3:GetObject）必須
Resource対象リソースのARN必須
Condition条件（IPアドレス制限、時間制限等）任意

ポリシー評価ロジック（超重要！）：

  1. デフォルトは全て暗黙的なDeny（拒否）
  2. 明示的なAllow（許可）があれば、暗黙的Denyを上書きして許可
  3. 明示的なDeny（拒否）は、どんなAllowよりも常に優先

つまり: Deny > Allow > デフォルトDeny

ポリシーの種類

種類説明管理者
AWS管理ポリシーAWSが事前に用意したポリシー（ReadOnlyAccess等）AWS
カスタマー管理ポリシーユーザーが独自に作成するポリシー。再利用可能ユーザー
インラインポリシー特定のユーザー/グループ/ロールに直接埋め込む。再利用不可ユーザー

問題パターン: 「複数のIAMユーザーに同じ権限を付与する最も効率的な方法は？」→ IAMグループを作成し、グループにポリシーをアタッチ。ユーザーをグループに追加する。個別にポリシーをアタッチするのは管理が煩雑。

2.3 IAMロール

IAMロールはAWSサービスやアプリケーションに権限を付与する仕組みです。IAMユーザーとの最大の違いは、長期的な認証情報（パスワードやアクセスキー）を持たない点です。一時的なセキュリティトークンを使用します。

ロールの主な用途

  • EC2インスタンスロール: EC2からS3にアクセスする際、アクセスキーをEC2に置くのではなく（危険！）、ロールをアタッチする
  • Lambdaの実行ロール: Lambda関数がDynamoDBに書き込む権限等
  • クロスアカウントアクセス: 別のAWSアカウントのリソースにアクセスする（AssumeRole）
  • 外部ID連携: GitHub Actions、SAML/OIDC連携

2.4 クロスアカウントアクセス（AssumeRole）

AssumeRoleは、あるAWSアカウント（またはサービス）が、別のアカウントのIAMロールを一時的に引き受ける仕組みです。一時的なセキュリティ認証情報（アクセスキー + シークレットキー + セッショントークン）が発行されます。

問題パターン: 「別のAWSアカウントのS3にアクセスするには？」→ クロスアカウントIAMロールを作成し、AssumeRoleで引き受ける。アクセスキーを共有するのはNG（セキュリティリスク）。

2.5 MFA（多要素認証）

パスワードだけでなく、物理デバイスやアプリが生成するワンタイムコードも要求する認証方式。ルートユーザーには必須で設定すべきです。

  • 仮想MFAデバイス: Google Authenticator、Authy等のスマホアプリ
  • U2Fセキュリティキー: YubiKey等の物理キー
  • ハードウェアMFAデバイス: Gemalto等の専用トークンデバイス

2.6 AWS Organizations と SCP

AWS Organizationsは、複数のAWSアカウントを一元管理するサービスです。大企業では開発用・本番用・セキュリティ用など複数のアカウントを使い分けます。

  • OU（Organizational Unit）: アカウントをグループ化する単位
  • SCP（Service Control Policy）: OU全体に適用する権限の上限。IAMポリシーよりも優先される（SCPでDenyされたらIAMでAllowしてもアクセス不可）
  • 一括請求: 全アカウントの利用料をまとめて支払い（ボリュームディスカウント適用）

問題パターン: 「開発アカウントでEC2の高額インスタンスタイプの起動を禁止するには？」→ SCPで該当インスタンスタイプのRunInstancesをDeny。SCPはIAMの上位にある「ガードレール」。

2.7 IAM Identity Center（旧AWS SSO）

複数のAWSアカウントやビジネスアプリケーション（Salesforce等）へのシングルサインオン（SSO = 1回のログインで複数サービスにアクセス）を実現するサービス。Active DirectoryやOkta等の外部IDプロバイダーとも連携可能。

2.8 IAM Access Analyzer

S3バケットやIAMロール等のリソースが外部（別アカウントやインターネット）からアクセス可能になっていないかを自動検出するサービス。意図しない公開設定を発見できます。

問題パターン: 「S3バケットが意図せず公開されていないか確認するには？」→ IAM Access Analyzerを使用。外部アクセスを検出し、セキュリティリスクを可視化。

ツリスポのGitHub ActionsがECRにDockerイメージをpushするとき、IAMロールを使っています。GitHub Actionsに直接アクセスキーを渡す方法もありますが、これはキー漏洩のリスクがあり危険です。代わりにOIDC（OpenID Connect）でIAMロールを引き受ける方式を採用。一時的な認証情報のみが使われ、長期キーがGitHub側に保存されません。

ツリスポのApp Runnerにはインスタンスロールがアタッチされています。このロールのポリシーには「ECRからのイメージpull権限」だけを付与（最小権限の原則）。S3への書き込み権限やRDSへのアクセス権限は含めません。必要になったら、その時に最小限の権限を追加します。

CLIコマンド例

# IAMユーザー一覧
aws iam list-users

# IAMユーザーにポリシーをアタッチ
aws iam attach-user-policy \
  --user-name dev-user \
  --policy-arn arn:aws:iam::aws:policy/AmazonS3ReadOnlyAccess

# IAMロール作成
aws iam create-role \
  --role-name TsurispotAppRole \
  --assume-role-policy-document file://trust-policy.json

# MFAデバイス一覧
aws iam list-mfa-devices --user-name dev-user

# パスワードポリシー設定
aws iam update-account-password-policy \
  --minimum-password-length 14 \
  --require-symbols \
  --require-numbers \
  --require-uppercase-characters \
  --max-password-age 90

第2章まとめ

  • ルートユーザーは日常使いしない。MFAを必ず設定。アクセスキーは作らない
  • IAMポリシーはJSON形式: Effect, Action, Resource, Condition
  • ポリシー評価: 明示的Deny > Allow > 暗黙的Deny
  • IAMロール = 一時的な認証情報。EC2やLambdaに権限を付与する安全な方法
  • AssumeRole = クロスアカウントアクセスの仕組み（STS経由）
  • AWS Organizations + SCP = 複数アカウントのガードレール
  • 最小権限の原則 = 必要最小限の権限だけを付与する

リソースが外部（別アカウントやインターネット）から意図せずアクセス可能になっていないかを自動検出する。

3 EC2（Elastic Compute Cloud）

EC2はAWSの最も基本的なコンピューティングサービスで、クラウド上に仮想サーバー（インスタンス）を立ち上げることができます。「必要な時に、必要なスペックのサーバーを、数分で起動できる」のがEC2の価値です。

パソコンの購入に例えると：

  • AMI = パソコンに入れるOS（Windows? Mac? Linux?）
  • インスタンスタイプ = パソコンのスペック（CPU、メモリ、どれくらい速い？）
  • EBS = ハードディスク（データを保存する場所、容量は？SSD? HDD?）
  • セキュリティグループ = ファイアウォール（どの通信を許可する？）
  • キーペア = パソコンのログインパスワード（SSH接続の鍵）

3.1 インスタンスタイプの命名規則

EC2のインスタンスタイプ名には規則があります。例えば m5.xlarge は：

  • m = ファミリー（汎用 = General Purpose）
  • 5 = 世代（第5世代、数字が大きいほど新しく高性能）
  • xlarge = サイズ（nano < micro < small < medium < large < xlarge < 2xlarge ...）

追加属性がつく場合もあります。例: m5a.xlarge の「a」はAMDプロセッサー、m6g.xlarge の「g」はAWS Graviton（ARM）プロセッサー。

主要なインスタンスファミリー

ファミリー最適な用途覚え方
T系 (T3, T3a, T4g)バースト可能な汎用。開発環境、小規模WebサーバーTurbo（バーストするから）
M系 (M5, M6i, M7g)バランスの取れた汎用。アプリサーバー、中規模DBMain（メイン＝汎用）
C系 (C5, C6i, C7g)コンピューティング最適化。バッチ処理、科学計算、ゲームサーバーCompute
R系 (R5, R6i, R7g)メモリ最適化。インメモリDB、リアルタイム分析RAM

T系インスタンスのバーストクレジット：

T系は「バースト」方式で動作します。通常はベースラインのCPU性能で動き、クレジットを蓄積。高負荷時にクレジットを消費して一時的に高速化。クレジットがなくなるとベースライン性能に戻ります。

  • T3 Unlimitedを有効にすると、クレジット枯渇後も高性能を維持（ただし追加課金あり）
  • 常に高負荷ならT系ではなくM系やC系を選ぶべき

問題パターン: 「メモリ集約型のワークロードに最適なインスタンスは？」→ R系（メモリ最適化）。「R = RAM」と覚える。

問題パターン: 「機械学習のトレーニングに最適なインスタンスは？」→ P系（GPU）。推論（inference）ならG系やInf系も可。

3.2 AMI（Amazon Machine Image）

AMIはEC2インスタンスのテンプレート（設計図）です。OS、ミドルウェア、アプリケーションの設定をまとめて保存したもので、AMIからインスタンスを起動すると、毎回同じ環境が再現されます。

  • AWS提供AMI: Amazon Linux 2023, Ubuntu, Windows Server等
  • Marketplace AMI: サードパーティが作成した有料/無料AMI
  • カスタムAMI: 自分で作成。自社アプリを入れた状態で保存し、複数のインスタンスに展開

AMIはリージョン固有ですが、別リージョンにコピーできます。

3.3 セキュリティグループ

EC2のインスタンスレベルの仮想ファイアウォールです。どのトラフィック（通信）を許可するかを制御します。

セキュリティグループの重要な特徴：

  • ステートフル: インバウンド（入り）を許可すると、そのレスポンス（出）は自動的に許可される
  • 許可ルールのみ: 「拒否」ルールは設定できない。デフォルトは全拒否で、許可したいものだけを追加
  • インスタンスレベル: 各インスタンスに個別にアタッチ
  • 複数のSGを1つのインスタンスにアタッチ可能（ルールは全て合算される）
  • 他のSGをソースに指定可能（例: ALBのSGからの通信のみ許可）

3.4 購入オプション（コスト最適化の要）

EC2のコスト最適化は試験の超頻出分野です。各購入オプションの特徴と使い分けを完全に理解しましょう。

問題パターン1: 「24時間365日稼働するWebサーバーのコスト最適化は？」→ リザーブドインスタンス or Savings Plans（長期契約で割引）

問題パターン2: 「中断されても問題ないバッチ処理のコスト最適化は？」→ スポットインスタンス（最大90%OFF、ただし中断リスクあり）

3.5 EBS（Elastic Block Store）

EBSはEC2にアタッチする永続的なブロックストレージ（仮想ハードディスク）です。EC2を停止・終了してもデータが保持されます（削除保護設定時）。

EBSボリュームタイプ

タイプ名称IOPS用途
gp3汎用SSD（最新）3,000〜16,000一般的なワークロード。最もコスパが良い
gp2汎用SSD（旧世代）容量に比例(最大16,000)gp3への移行推奨
io2/io1プロビジョンドIOPS SSD最大64,000 (io2 Block Express: 256,000)高性能DB。ミッションクリティカル
st1スループット最適化HDD500 IOPSビッグデータ、ログ処理
sc1コールドHDD250 IOPSアクセス頻度の低いデータ。最安

EBSの重要な特徴：

  • EBSは特定のAZに固定される。別AZのEC2にはアタッチ不可（スナップショット経由で移動可能）
  • スナップショット: EBSのバックアップ。S3に保存される。増分バックアップ（変更分のみ保存）
  • 暗号化: AES-256。KMSキーを使用。暗号化は無料
  • マルチアタッチ: io1/io2のみ、同じAZの最大16のEC2に同時アタッチ可能

3.6 インスタンスストア

EC2ホストマシンに物理的に接続された一時的なストレージ。EBSよりも高速（特にI/O集約型タスク）ですが、インスタンス停止・終了でデータが消失します。キャッシュやバッファに最適。

問題パターン: 「高IOPSが必要だが、データ消失が許容されるワークロードに最適なストレージは？」→ インスタンスストア。永続性が必要ならio2 EBS。

ツリスポをEC2で運用する場合: t3.medium（2vCPU, 4GB RAM）で月間5万PVまで対応可能。月額約$30。gp3 EBS 20GBをアタッチ（月額約$1.6）。月間50万PVになったらm6i.large（2vCPU, 8GB RAM）にスケールアップし、リザーブドインスタンス3年全前払いで年間コスト約40%削減。

現在のApp Runnerは内部的にFargate（EC2の代わりにAWSが管理するコンテナ実行環境）で動いています。EC2の場合と違い、OS管理・パッチ適用・スケーリング設定が不要。ただしEC2ほど細かいチューニング（インスタンスタイプ選択、EBSタイプ指定等）はできません。

CLIコマンド例

# EC2インスタンスの起動
aws ec2 run-instances \
  --image-id ami-0abcdef1234567890 \
  --instance-type t3.medium \
  --key-name my-key \
  --security-group-ids sg-0123456789abcdef0 \
  --subnet-id subnet-0123456789abcdef0

# インスタンス一覧
aws ec2 describe-instances --query 'Reservations[*].Instances[*].[InstanceId,State.Name,InstanceType]' --output table

# EBSボリューム作成
aws ec2 create-volume \
  --volume-type gp3 \
  --size 100 \
  --availability-zone ap-northeast-1a

# スナップショット作成
aws ec2 create-snapshot --volume-id vol-0123456789abcdef0 --description "Daily backup"

第3章まとめ

  • インスタンスタイプ: ファミリー + 世代 + サイズ（例: m5.xlarge）
  • T系=バースト汎用、M系=汎用、C系=コンピュート、R系=メモリ、I/D系=ストレージ、P/G系=GPU
  • AMI = インスタンスのテンプレート。OS + ミドルウェアのスナップショット
  • SG = ステートフルなファイアウォール。許可ルールのみ
  • 購入: オンデマンド（定価） < Savings Plans/RI（最大72%OFF） < スポット（最大90%OFF、中断あり）
  • EBS: gp3が最もコスパ良い。io2は高性能DB向け。st1/sc1はHDD
  • EBSはAZ固定。スナップショットでバックアップ（S3に保存、増分）
  • インスタンスストア = 一時的だが超高速。停止でデータ消失

Q1. メモリ最適化インスタンスのファミリーは？

答えを見る

R系（R = RAM）。超大容量メモリが必要ならX系。

Q2. 中断されても問題ないバッチ処理に最適な購入オプションは？

答えを見る

スポットインスタンス。最大90%OFFだが、AWSが2分前通知で中断する可能性あり。

Q3. EBSボリュームを別のAZに移動するには？

答えを見る

スナップショットを取得し、それを別のAZで新しいEBSボリュームとして復元する。EBSは直接AZ間移動できない。

Q4. ブートボリュームに使えないEBSタイプは？

答えを見る

st1（スループットHDD）とsc1（コールドHDD）。ブートボリュームはSSD（gp2/gp3/io1/io2）のみ。

4 ストレージサービス

AWSのストレージサービスの中でも、S3（Simple Storage Service）は試験で最も出題頻度が高いサービスの一つです。S3だけで10問以上出ることもあります。完全に理解しましょう。

ストレージの種類を倉庫に例えると：

  • S3（オブジェクトストレージ） = 貸し倉庫。モノ（ファイル）を箱に入れてラベル（キー）を付けて保管。中身を部分的に書き換えるのは不得意。丸ごと出し入れ向き。
  • EBS（ブロックストレージ） = 自分の机の引き出し。高速にアクセスでき、中身の一部だけ書き換え可能。ただし1つのPC（EC2）にしか接続できない。
  • EFS（ファイルストレージ） = 共有フォルダ。複数のPC（EC2）から同時に読み書きできる。

4.1 S3の基本概念

S3はオブジェクトストレージサービスで、画像、動画、バックアップ、ログなどあらゆるファイルを保存できます。容量は実質無制限で、1オブジェクトあたり最大5TBまで。

バケットとオブジェクト

  • バケット: ファイルを保存するコンテナ。名前はグローバルに一意（世界中で重複不可）
  • オブジェクト: バケット内のファイル。キー（パス名）、データ本体、メタデータで構成
  • キーの例: images/spots/akashi-port.jpg
  • S3に「フォルダ」という概念は実は存在しない。キー名にスラッシュを含めることで、コンソール上ではフォルダのように見える

バケットの命名規則

  • 3〜63文字。小文字、数字、ハイフンのみ
  • 先頭は英小文字 or 数字
  • IPアドレス形式（192.168.1.1等）は不可
  • グローバルに一意であること

4.2 S3ストレージクラス（超重要！）

S3にはデータのアクセス頻度に応じた7つのストレージクラスがあり、使い分けによってコストを最適化できます。

ストレージクラスアクセス頻度最小保存期間取り出しコスト用途
S3 Standard頻繁なしなしアクティブなデータ
S3 Intelligent-Tiering不明/変動なしなしアクセスパターンが読めないデータ
S3 Standard-IA月1回程度30日ありバックアップ、DR
S3 One Zone-IA月1回程度30日あり再作成可能なデータ（サムネイル等）
Glacier Instant Retrieval四半期1回90日あり即時アクセスが必要なアーカイブ
Glacier Flexible Retrieval年1-2回90日あり（分〜時間）アーカイブ（取り出しに時間OK）
Glacier Deep Archive年1回未満180日あり（12時間）法令遵守の長期保存。最安

問題パターン1: 「アクセスパターンが予測できないデータに最適なストレージクラスは？」→ S3 Intelligent-Tiering（自動的に最適なクラスに移行、取り出し料金なし）

問題パターン2: 「法令で7年間保存が必要だが、ほぼアクセスしないデータは？」→ Glacier Deep Archive（最安、取り出しに12時間、180日最小保存期間）

問題パターン3: 「再作成可能なサムネイル画像を安く保存したい」→ S3 One Zone-IA（1つのAZのみに保存するため安い、データ消失リスクはあるが再作成可能）

4.3 S3ライフサイクルポリシー

オブジェクトを自動的にストレージクラス間で移行したり、削除するルールを設定できます。

4.4 S3の重要な機能

バージョニング

同じキーのオブジェクトの全バージョンを保持します。誤って上書き・削除しても復元可能。バケットレベルで有効化します。

暗号化

方式鍵の管理説明
SSE-S3AWS（S3が管理）デフォルトの暗号化。AES-256。最も手軽
SSE-KMSAWS KMS鍵の使用履歴をCloudTrailで監査可能。権限管理が細かい
SSE-Cユーザーが提供リクエストごとに鍵を送信。HTTPS必須
クライアント側暗号化ユーザーが完全管理アップロード前にクライアントで暗号化

レプリケーション

  • CRR（Cross-Region Replication）: 別リージョンに自動コピー。DR、低レイテンシーアクセス
  • SRR（Same-Region Replication）: 同リージョン内の別バケットに自動コピー。ログ集約等
  • バージョニングの有効化が前提条件

その他の重要機能

  • S3 Transfer Acceleration: CloudFrontエッジロケーション経由で高速アップロード
  • プレサインドURL: 一時的なアクセス許可URL。有効期限付き。非公開オブジェクトの一時共有に便利
  • S3 Select / Glacier Select: SQLライクなクエリでオブジェクト内の一部データだけ取得（全データ取得より安く速い）
  • イベント通知: オブジェクトの作成/削除をトリガーに Lambda/SQS/SNS/EventBridge に通知
  • S3 Object Lock: WORM（Write Once Read Many）モデル。規制遵守のため、一度書いたら変更/削除不可
  • MFA Delete: 削除操作にMFA認証を要求。バージョニング有効化が前提
  • Multipart Upload: 100MB以上のファイルは分割アップロードを推奨。5GBを超えるファイルは必須

問題パターン: 「規制により、一度保存したデータを一定期間変更・削除不可にしたい」→ S3 Object Lock（WORMモデル）。Glacierの Vault Lock も同様。

問題パターン: 「大量のクライアントが世界中からS3に大きなファイルをアップロードする際の高速化」→ S3 Transfer Acceleration + Multipart Upload

4.5 EFS（Elastic File System）

Linux用のマネージドNFS（Network File System）。複数のEC2インスタンスから同時にマウント（接続）できるのが最大の特徴。EBSは基本1対1だが、EFSは多対多。

  • POSIX互換: Linux標準のファイルシステム
  • 自動スケーリング: 容量を事前指定する必要なし（使った分だけ課金）
  • マルチAZ: デフォルトで複数AZにデータを複製
  • ストレージクラス: Standard / Infrequent Access（IA）。ライフサイクル管理あり

4.6 FSx

  • FSx for Windows File Server: Windows向けのマネージドファイルサーバー。SMBプロトコル、Active Directory統合
  • FSx for Lustre: 高性能計算（HPC）向け。機械学習、動画処理。S3と統合可能
  • FSx for NetApp ONTAP: マルチプロトコル（NFS/SMB/iSCSI）
  • FSx for OpenZFS: ZFSファイルシステム

4.7 Storage Gateway

オンプレミス環境とAWSストレージを統合するハイブリッドストレージサービス。

種類説明用途
S3 File GatewayNFS/SMBでアクセスし、裏でS3に保存ファイル共有のクラウド移行
Volume GatewayiSCSIでブロックストレージ提供。Stored/Cachedオンプレミスのブロックストレージバックアップ
Tape Gateway仮想テープライブラリ(VTL)をS3/Glacierにバックアップテープバックアップの置き換え

4.8 Snow Family

大容量データの物理的な移行に使うデバイス。ネットワーク転送では何週間もかかるデータを、デバイスに入れてAWSに郵送します。

デバイス容量用途
Snowcone8TB HDD / 14TB SSD小規模データ移行、エッジコンピューティング
Snowball Edge80TB（Storage Optimized）大規模データ移行、エッジ処理
Snowmobile100PB超大規模移行（トラック1台分！）

問題パターン: 「10PBのデータをAWSに移行。ネットワークでは3ヶ月かかる」→ Snowball Edgeを複数台使用。100PBならSnowmobile。

ツリスポの画像ファイル（スポット写真、魚の写真、OGP画像）をS3 Standardに保存。頻繁にアクセスされる人気スポットの画像はStandard、3ヶ月アクセスのない古い写真はライフサイクルポリシーでS3 Standard-IAに自動移行。CloudFront経由で配信すれば日本全国どこからでも高速アクセス。コスト試算: 画像10GB = S3 Standard: 月$0.23 + CloudFront配信コスト。

ツリスポで将来ユーザーが釣果写真をアップロードする機能を追加する場合: プレサインドURLを発行してブラウザから直接S3にアップロード（サーバーを経由しないので高速・低負荷）。アップロード後にS3イベント通知でLambdaを起動し、画像のリサイズとサムネイル生成を自動実行。

CLIコマンド例

# バケット作成
aws s3 mb s3://tsurispot-images-prod --region ap-northeast-1

# ファイルアップロード
aws s3 cp ./images/spots/ s3://tsurispot-images-prod/spots/ --recursive

# バケット内のオブジェクト一覧
aws s3 ls s3://tsurispot-images-prod/spots/ --human-readable

# ライフサイクルポリシー設定
aws s3api put-bucket-lifecycle-configuration \
  --bucket tsurispot-images-prod \
  --lifecycle-configuration file://lifecycle.json

# プレサインドURL発行（1時間有効）
aws s3 presign s3://tsurispot-images-prod/spots/photo.jpg --expires-in 3600

第4章まとめ

  • S3 = オブジェクトストレージ。容量無制限、1オブジェクト最大5TB
  • 7つのストレージクラス: Standard > Intelligent-Tiering > Standard-IA > One Zone-IA > Glacier Instant > Glacier Flexible > Deep Archive
  • ライフサイクルポリシーでストレージクラスを自動移行（コスト最適化）
  • 暗号化: SSE-S3(デフォルト) / SSE-KMS(監査可能) / SSE-C(ユーザー鍵) / クライアント側
  • レプリケーション: CRR(クロスリージョン) / SRR(同一リージョン)。バージョニング必須
  • EFS = Linux NFS共有。複数EC2から同時マウント。自動スケーリング
  • FSx = Windows(SMB)やHPC(Lustre)向けマネージドファイルシステム
  • Storage Gateway = オンプレミスとAWSストレージのハイブリッド接続
  • Snow Family = 大容量データの物理移行（Snowcone/Snowball/Snowmobile）

Q1. アクセスパターンが予測できないデータに最適なS3ストレージクラスは？

答えを見る

S3 Intelligent-Tiering。自動的にアクセス頻度に基づいてクラスを移行。取り出し料金なし。

Q2. S3レプリケーションの前提条件は？

答えを見る

ソース・ターゲット両方のバケットでバージョニングが有効であること。

Q3. 複数のEC2から同時にファイル共有したい（Linux環境）。最適なサービスは？

答えを見る

EFS（Elastic File System）。EBSは基本1つのEC2にしかアタッチできない（io1/io2のマルチアタッチは例外だが同一AZ限定）。

Q4. 5TBのファイルをS3にアップロードする方法は？

答えを見る

Multipart Uploadを使用。5GBを超えるファイルはMultipart Upload必須。100MB以上で推奨。

5 VPCとネットワーキング

VPC（Virtual Private Cloud）はAWSの中に自分専用のプライベートネットワーク空間を作るサービスです。初心者が最もつまずきやすい分野ですが、SAA試験では非常に重要。ここを丁寧に理解しましょう。

VPCを「家」に例えると：

  • VPC = あなたの家（敷地全体）。外の世界とは壁で区切られている。
  • サブネット = 家の中の部屋（リビング、寝室、書斎）。用途ごとに分ける。
  • インターネットゲートウェイ（IGW） = 玄関ドア。外の世界（インターネット）との出入口。
  • NATゲートウェイ = 郵便受け。中から外へは手紙を出せるが、外から勝手に入ることはできない。
  • ルートテーブル = 道案内の看板。「この部屋に行くにはこの廊下を通れ」
  • セキュリティグループ = 各部屋の鍵。誰を入れるか部屋ごとに決める。
  • NACL = 門番。家の入口で出入りをチェックする。

5.1 IPアドレスの基礎

VPCを理解するには、まずIPアドレスの基本を知る必要があります。

IPアドレスは、ネットワーク上のデバイスの住所です。IPv4アドレスは「192.168.1.10」のように、0〜255の数字4つをドットで区切った形式で表されます。

CIDR（Classless Inter-Domain Routing）表記

CIDRはIPアドレスの範囲を表す書き方です。例：10.0.0.0/16

  • /16 の部分は「ネットワーク部分が先頭16ビット」を意味する
  • 残りのビット（32-16=16ビット）がホスト部分 → 2^16 = 65,536個のIPアドレスが使える
CIDRIPアドレス数用途の目安
/1665,536個VPC全体（最大サイズ）
/204,096個大きめのサブネット
/24256個一般的なサブネット
/2816個小さなサブネット（最小サイズ）

AWSがサブネットごとに予約するIPアドレス（5個）：

例: サブネット 10.0.1.0/24（256個）の場合、実際に使えるのは251個。

  • 10.0.1.0 : ネットワークアドレス
  • 10.0.1.1 : VPCルーター用
  • 10.0.1.2 : DNS用
  • 10.0.1.3 : 将来の利用用に予約
  • 10.0.1.255 : ブロードキャストアドレス

5.2 VPC、サブネット、ルートテーブル

VPCの作成

VPCはリージョン単位で作成します。1つのVPCは1つのリージョン内に存在し、複数のAZにまたがることができます。CIDRブロックは/16〜/28の範囲で指定します。

パブリックサブネットとプライベートサブネット

  • パブリックサブネット: ルートテーブルにインターネットゲートウェイ（IGW）へのルートがあるサブネット。Webサーバーやロードバランサーを配置
  • プライベートサブネット: IGWへのルートがないサブネット。データベースやアプリケーションサーバーを配置。外部から直接アクセス不可

インターネットゲートウェイ（IGW）

VPCとインターネットを接続するゲートウェイ。VPCに1つだけアタッチ可能。IGWがないとVPC内からインターネットに出られません。高可用性で冗長化済み（AWS管理）。

NATゲートウェイ / NATインスタンス

プライベートサブネットのインスタンスがインターネットに出るためのサービス（例: OSパッチのダウンロード、npm install等）。外からプライベートサブネットへの直接接続は引き続き不可。

NATゲートウェイ（推奨）NATインスタンス（レガシー）
管理AWSマネージドユーザーが管理
可用性AZ内で自動冗長化自分で冗長化設定
帯域幅最大100Gbpsインスタンスタイプ依存
コスト時間+データ処理量課金インスタンス費用のみ（安い場合あり）
SG不要必要

問題パターン: 「プライベートサブネットのEC2がインターネットからパッケージをダウンロードするには？」→ パブリックサブネットにNATゲートウェイを配置し、プライベートサブネットのルートテーブルにNAT GWへのルートを追加。

ルートテーブル

通信先を決めるルール表。サブネットごとにルートテーブルを関連付けます。

# パブリックサブネットのルートテーブル例
送信先              ターゲット
10.0.0.0/16        local       (VPC内通信)
0.0.0.0/0          igw-xxxxx   (その他は全てIGWへ = インターネット)

# プライベートサブネットのルートテーブル例
送信先              ターゲット
10.0.0.0/16        local       (VPC内通信)
0.0.0.0/0          nat-xxxxx   (その他は全てNAT GWへ)

5.3 セキュリティグループ vs NACL

VPCには2種類のファイアウォールがあり、それぞれの違いは試験で必ず出題されます。

セキュリティグループ（SG）ネットワークACL（NACL）
レベルインスタンスレベルサブネットレベル
ステートステートフル（戻りの通信は自動許可）ステートレス（行きと戻り両方のルール必要）
ルール許可ルールのみ許可 + 拒否ルール
評価全ルールを評価してから判定番号順に評価（最初にマッチしたルールで決定）
デフォルトインバウンド: 全拒否 / アウトバウンド: 全許可インバウンド・アウトバウンドとも全許可

問題パターン1: 「特定のIPアドレスからのアクセスを拒否したい」→ NACL（拒否ルールが使えるのはNACLだけ。SGは許可ルールのみ）

問題パターン2: 「インバウンドを許可したら、レスポンスも自動で許可されるのは？」→ SG（ステートフル）。NACLはステートレスなので戻りの通信も明示的に許可が必要。

5.4 VPCピアリングとTransit Gateway

VPCピアリング

2つのVPCをプライベートに接続します。異なるリージョン、異なるアカウント間でも可能。

VPCピアリングは推移的ではない！ A-B間とB-C間がピアリングされていても、A-Cは直接通信できない。A-C間にも別途ピアリングが必要。これは試験で頻出。

Transit Gateway

VPCピアリングは「1対1」接続。VPCが多数あるとピアリングだらけで複雑に。Transit Gatewayはハブ型で、1つのTGWに全VPCを接続すれば全VPC間で通信可能。推移的ルーティングをサポート。

5.5 VPCエンドポイント

VPCからS3やDynamoDB等のAWSサービスにインターネットを経由せずプライベートに接続する仕組み。セキュリティ向上とデータ転送コスト削減。

種類対象サービス仕組みコスト
ゲートウェイエンドポイントS3, DynamoDB のみルートテーブルにエントリ追加無料
インターフェースエンドポイントその他多数のサービスENI（PrivateLink）をサブネットに配置時間+データ課金

問題パターン: 「プライベートサブネットのEC2からS3にインターネットを経由せずアクセスしたい」→ S3ゲートウェイエンドポイント（無料！）。NATゲートウェイ経由でもアクセスはできるが、データ転送料金がかかる。

5.6 VPN接続とDirect Connect

Site-to-Site VPN

オンプレミスのネットワークとVPCを暗号化されたインターネット経由のトンネルで接続。数分で構築可能。

  • VPC側: Virtual Private Gateway (VGW) をアタッチ
  • オンプレ側: Customer Gateway (CGW) を設定
  • IPsec暗号化、冗長化のため2つのトンネルが自動作成

AWS Direct Connect

AWSとオンプレミスを専用線（物理的な光ファイバー）で接続。低レイテンシー、高帯域幅、安定した通信。構築に数週間〜数ヶ月かかる。

  • 帯域幅: 1Gbps / 10Gbps / 100Gbps
  • VPNよりも高額だが、大容量・安定通信に優れる
  • 暗号化はデフォルトでは行われない（Direct Connect + VPNの組み合わせで暗号化可能）

問題パターン: 「大量のデータをオンプレミスからAWSに安定・高速に転送したい」→ Direct Connect。すぐに必要ならSite-to-Site VPN（暫定）+ Direct Connect（本番）の組み合わせ。

5.7 VPCフローログ

VPC内のネットワークインターフェースを通過するIPトラフィックのログを記録します。CloudWatch LogsまたはS3に保存。セキュリティ監査やトラブルシューティングに使用。

ツリスポのApp RunnerはAWSが管理するVPC内で動いていますが、もしEC2で構築するなら以下の構成：

  • VPC: 10.0.0.0/16
  • パブリックサブネット: 10.0.1.0/24（AZ-a）と10.0.3.0/24（AZ-c）にALB配置
  • プライベートサブネット: 10.0.2.0/24（AZ-a）と10.0.4.0/24（AZ-c）にEC2とRDS
  • NATゲートウェイ経由でEC2がnpm install等のインターネットアクセス
  • S3へはゲートウェイエンドポイント（無料）でプライベート接続
  • 外部ユーザーはALB経由でのみアクセス可能（EC2に直接アクセス不可）

ツリスポが将来microCMSではなく自前のヘッドレスCMSをEC2上で運用する場合、CMSのEC2はプライベートサブネットに配置。管理画面へのアクセスはClient VPN経由。データベースはプライベートサブネットのRDS Aurora Serverless。VPCフローログでセキュリティ監視。

CLIコマンド例

# VPC作成
aws ec2 create-vpc --cidr-block 10.0.0.0/16 --tag-specifications 'ResourceType=vpc,Tags=[{Key=Name,Value=tsurispot-vpc}]'

# サブネット作成
aws ec2 create-subnet --vpc-id vpc-xxxxx --cidr-block 10.0.1.0/24 --availability-zone ap-northeast-1a

# インターネットゲートウェイ作成・アタッチ
aws ec2 create-internet-gateway
aws ec2 attach-internet-gateway --internet-gateway-id igw-xxxxx --vpc-id vpc-xxxxx

# NATゲートウェイ作成（Elastic IPが必要）
aws ec2 allocate-address --domain vpc
aws ec2 create-nat-gateway --subnet-id subnet-public --allocation-id eipalloc-xxxxx

# VPCエンドポイント（S3ゲートウェイ型）
aws ec2 create-vpc-endpoint --vpc-id vpc-xxxxx --service-name com.amazonaws.ap-northeast-1.s3 --route-table-ids rtb-xxxxx

第5章まとめ

  • VPC = AWS内の自分専用ネットワーク。リージョン単位、複数AZにまたがる
  • CIDR: /16=65,536IP、/24=256IP。AWSはサブネットごとに5つ予約
  • パブリックサブネット = IGWへのルートあり。プライベートサブネット = なし
  • NATゲートウェイ: プライベート → インターネットの片方向通信を実現
  • SG = ステートフル・許可のみ・インスタンスレベル。NACL = ステートレス・許可+拒否・サブネットレベル
  • VPCピアリングは推移的でない（A-B, B-CでA-C不可）。Transit Gatewayはハブ型
  • VPCエンドポイント: S3/DynamoDBはゲートウェイ型（無料）、その他はインターフェース型
  • Direct Connect = 専用線（高速安定、構築に時間）。VPN = 暗号化トンネル（すぐ構築可能）

Q1. 特定のIPアドレスからのアクセスを「拒否」するにはSGとNACLどちらを使う？

答えを見る

NACL。セキュリティグループには拒否ルールがない（許可ルールのみ）。

Q2. VPCピアリングは推移的か？

答えを見る

推移的ではない。A-B, B-C間がピアリングされていても、A-Cは直接通信不可。別途A-C間のピアリングが必要。多数のVPCにはTransit Gatewayを使う。

Q3. プライベートサブネットからS3にインターネット経由せずアクセスするには？

答えを見る

S3ゲートウェイエンドポイント（無料）。ルートテーブルにエントリが追加される。

Q4. Direct Connectのデフォルトで暗号化は行われる？

答えを見る

いいえ。暗号化が必要な場合はDirect Connect上にVPNを構築する。

Q5. NATゲートウェイとNATインスタンスの主な違いは？

答えを見る

NATゲートウェイはAWSマネージドで自動冗長化、最大100Gbps。NATインスタンスはユーザー管理でSGも必要だが、場合によってはコストが安い。AWS公式はNATゲートウェイ推奨。

6 データベースサービス

AWSには用途に応じた多数のデータベースサービスがあります。SAA試験ではRDS vs DynamoDB の使い分けが最頻出。「どんなデータにどのDBを使うか」を判断できるようになりましょう。

データベースの種類を整理棚に例えると：

  • RDS（リレーショナルDB） = Excel表。行と列がきっちり決まっている。「社員名簿」のように構造化されたデータ向き。
  • DynamoDB（NoSQL） = 付箋メモの箱。自由な形式のデータを高速に出し入れ。「いいね数」「セッション情報」等の柔軟なデータ向き。
  • ElastiCache = 手元のメモ帳（キャッシュ）。よく使うデータを一時的に書き留めておいて超高速にアクセス。
  • Redshift = 巨大な集計表。大量のデータを集計・分析する専用ツール。

6.1 リレーショナルDB vs NoSQL

リレーショナルDB（RDS, Aurora）NoSQL（DynamoDB）
データ構造行と列のテーブル（スキーマ固定）キーバリュー / ドキュメント（スキーマ柔軟）
クエリSQL（複雑なJOINが可能）プライマリキーベース（JOINは不得意）
スケーリング垂直（インスタンスサイズアップ）水平（パーティション自動分割）
トランザクションACID準拠結果整合性（強い整合性も選択可）
最適な用途ユーザー情報、注文管理、在庫管理セッション管理、IoTデータ、ゲームリーダーボード

6.2 RDS（Relational Database Service）

RDSはAWSが提供するマネージドなリレーショナルデータベースです。OSパッチ、バックアップ、フェイルオーバーをAWSが管理するため、運用負荷が大幅に軽減されます。

対応エンジン

  • MySQL / MariaDB / PostgreSQL : オープンソース系
  • Oracle / SQL Server : 商用DB
  • Amazon Aurora : AWS独自開発（MySQL/PostgreSQL互換で高性能）

マルチAZデプロイ（高可用性）

プライマリDBのデータを別のAZのスタンバイに同期レプリケーション（リアルタイム複製）。プライマリに障害が起きると、自動的にスタンバイにフェイルオーバー（切り替え）します。通常60〜120秒で復旧。

  • スタンバイは読み取りには使えない（フェイルオーバー専用）
  • 同じリージョン内の別AZ
  • DNSエンドポイントは変わらない（アプリの接続先変更不要）

リードレプリカ（読み取りスケーリング）

プライマリDBのデータを非同期レプリケーションで複製した読み取り専用DB。読み取り負荷をレプリカに分散します。

  • 最大15個のリードレプリカを作成可能（Aurora）。RDSは最大5個
  • 別リージョンにも作成可能（クロスリージョンリードレプリカ）
  • 読み取り専用なので書き込みはプライマリのみ
  • レプリカをプライマリに昇格（プロモーション）可能

問題パターン1: 「DB障害時に自動で切り替える仕組みは？」→ マルチAZデプロイ（同期レプリケーション + 自動フェイルオーバー）

問題パターン2: 「読み取り負荷を分散したい」→ リードレプリカを追加し、読み取りクエリをレプリカに向ける

問題パターン3: 「マルチAZは読み取りスケーリングに使える？」→ いいえ。スタンバイは読み取りに使えない（フェイルオーバー専用）。読み取りスケーリングはリードレプリカ。

RDS Proxy

Lambda関数からRDSに接続する際の問題を解決するサービス。Lambda は同時に何百もの関数が起動するため、DB接続が爆発的に増える。RDS Proxyがコネクションプーリング（接続を使い回す）を行い、DBの負荷を軽減します。

6.3 Amazon Aurora

AuroraはAWSが独自開発したリレーショナルDB。MySQL互換またはPostgreSQL互換で動作し、標準のMySQL/PostgreSQLより最大5倍高速。

Auroraの特長

  • ストレージ: 10GB〜128TB自動拡張。6コピーを3AZに分散 = 18コピーのデータ冗長性
  • リードレプリカ: 最大15個。フェイルオーバーの優先度設定可能
  • 自動フェイルオーバー: 通常30秒以内（RDSの60-120秒より高速）
  • バックトラック: DBを過去の任意の時点に巻き戻し（MySQL互換のみ）
  • クローニング: 本番DBのコピーを高速作成（テスト環境構築に便利）

Aurora Serverless v2

必要に応じて自動でスケールアップ/ダウンするAurora。アクセスがない時は最小キャパシティに縮小し、コスト削減。アクセスパターンが予測できないワークロードに最適。ACU（Aurora Capacity Unit）単位で課金。

Aurora Global Database

プライマリリージョンから他リージョンへ1秒未満のレイテンシーでレプリケーション。DR（災害復旧）に最適。RPO（Recovery Point Objective = データ損失許容量）は1秒。

問題パターン: 「MySQL互換で最高のパフォーマンスと可用性が必要」→ Aurora。「さらにグローバルDRが必要」→ Aurora Global Database（RPO 1秒）。

6.4 DynamoDB

DynamoDBはフルマネージドのNoSQLデータベース。サーバーの管理が一切不要で、単一桁ミリ秒（1〜9ms）のレイテンシーを実現します。

基本概念

  • テーブル: データを格納する単位
  • 項目（Item）: テーブル内の1レコード（最大400KB）
  • 属性（Attribute）: 項目内の各フィールド
  • パーティションキー: 必須の主キー。データの分散先を決定
  • ソートキー: オプション。パーティションキーと組み合わせて複合キーを作成

キャパシティモード

モード特徴最適な用途
オンデマンド使った分だけ課金。キャパシティ設定不要アクセスパターンが予測できない、新サービス
プロビジョニング事前にRCU/WCUを設定。Auto Scaling併用可アクセスパターンが安定しているサービス

セカンダリインデックス

  • GSI（グローバルセカンダリインデックス）: 元テーブルとは異なるパーティションキー/ソートキーでクエリ可能。テーブル作成後にも追加可能
  • LSI（ローカルセカンダリインデックス）: 同じパーティションキーで別のソートキーを使用。テーブル作成時にのみ定義可能

DynamoDB Streams

テーブルへの変更（INSERT/UPDATE/DELETE）をストリームとしてキャプチャ。Lambda関数をトリガーして変更に反応する処理が可能。

DAX（DynamoDB Accelerator）

DynamoDB用のインメモリキャッシュ。読み取りレイテンシーをミリ秒からマイクロ秒に短縮。DynamoDBのAPIと互換性があるため、コード変更が最小限。

DynamoDB Global Tables

複数リージョンでのマルチアクティブレプリケーション。どのリージョンでも読み書き可能。DynamoDB Streamsの有効化が前提。

問題パターン: 「ミリ秒のレイテンシーで大量の読み書きが必要なキーバリュー型データ」→ DynamoDB

問題パターン: 「DynamoDBの読み取りレイテンシーをさらに改善したい」→ DAX（マイクロ秒レベルのキャッシュ）

6.5 ElastiCache

マネージドなインメモリキャッシュサービス。DBの前段に置いて、頻繁にアクセスされるデータをメモリに保持し超高速（マイクロ秒〜ミリ秒）でレスポンスします。

RedisMemcached
データ構造文字列、リスト、セット、ハッシュ等（リッチ）文字列のみ（シンプル）

キャッシュ戦略：

  • Lazy Loading（遅延読み込み）: キャッシュにデータがなければDBから取得し、キャッシュに保存。キャッシュミスが発生するが、不要なデータをキャッシュしない
  • Write-Through（書き込み時キャッシュ）: DBに書き込む時に同時にキャッシュにも書く。常に最新だが、書き込みが遅い
  • TTL（Time to Live）: キャッシュの有効期限。古いデータが残るのを防ぐ

問題パターン: 「Redis vs Memcachedの違いは？」→ Redis: リッチなデータ構造、永続性、レプリケーション、マルチAZ。Memcached: シンプル、マルチスレッド。リーダーボードやセッション管理ならRedis。

第6章まとめ

  • RDS = マネージドなリレーショナルDB。MySQL/PostgreSQL/Oracle/SQL Server/Aurora対応
  • マルチAZ = 同期レプリケーション + 自動フェイルオーバー（高可用性、読み取りは不可）
  • リードレプリカ = 非同期レプリケーション。読み取り負荷分散（最大15個 for Aurora）
  • Aurora = AWS独自。MySQL/PgSQL互換で最大5倍高速。6コピー3AZ。Serverless v2で自動スケール
  • DynamoDB = フルマネージドNoSQL。ミリ秒レイテンシー。DAXでマイクロ秒に高速化
  • ElastiCache = インメモリキャッシュ。Redis（リッチ機能）vs Memcached（シンプル）
  • Redshift = データウェアハウス（大量データ分析）。Neptune = グラフDB。QLDB = 台帳DB
  • RDS Proxy = Lambda→RDSのコネクションプーリング

Q1. RDSマルチAZのスタンバイは読み取りに使える？

答えを見る

使えない。スタンバイはフェイルオーバー専用。読み取り分散にはリードレプリカを使う。

Q2. DynamoDBのレイテンシーをさらに改善するサービスは？

答えを見る

DAX（DynamoDB Accelerator）。読み取りレイテンシーをミリ秒からマイクロ秒に短縮するインメモリキャッシュ。

Q3. Auroraのストレージは何コピー何AZに分散？

答えを見る

6コピーを3つのAZに分散（各AZに2コピー = 合計6コピー）。非常に高い耐久性。

Q4. ElastiCacheでセッション管理やリーダーボードに適しているのは？

答えを見る

Redis。リッチなデータ構造（Sorted Set等）、永続性、レプリケーション、マルチAZをサポート。Memcachedはシンプルなキャッシュ専用。

Q5. 金融取引の改ざん不可能な変更履歴を記録するのに最適なDBは？

答えを見る

QLDB（Quantum Ledger Database）。不変で暗号的に検証可能な変更履歴（台帳）を提供。
