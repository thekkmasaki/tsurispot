13 モニタリングと運用 〜システムの健康状態を監視する〜

AWSで本番環境を運用する上で最も重要なことの一つが「モニタリング（監視）」です。サーバーが落ちていることに気づかない、CPU使用率が100%に張り付いているのに誰も知らない――そんな事態を防ぐのが、この章で学ぶサービス群です。

モニタリングは「車のダッシュボード」に例えられます。速度計、燃料計、エンジン警告灯など、車の状態を一目で確認できますよね。AWSのモニタリングサービスは、システム全体のダッシュボードを提供し、問題があれば警告を鳴らしてくれます。

13.1 CloudWatch 〜AWSの統合監視サービス〜

Amazon CloudWatchは、AWSリソースとアプリケーションの監視サービスです。メトリクス（数値データ）の収集、ログの管理、アラームの設定など、監視に関するほぼすべてを担います。

13.1.1 メトリクス（Metrics）

メトリクスとは、時系列で記録される数値データのことです。「EC2のCPU使用率」「ALBのリクエスト数」「RDSの接続数」などがメトリクスにあたります。

標準メトリクス vs カスタムメトリクス

種類説明例費用
標準メトリクスAWSが自動で収集EC2: CPUUtilization, NetworkIn/Out
ALB: RequestCount, TargetResponseTime		無料
カスタムメトリクスユーザーが独自に送信メモリ使用率、ディスク使用率、アプリ固有の数値有料（$0.30/メトリクス/月）

EC2の「メモリ使用率」は標準メトリクスに含まれない！ EC2のCPU使用率は自動で取得できますが、メモリ使用率はOSの中の情報なのでCloudWatch Agentをインストールしてカスタムメトリクスとして送信する必要があります。これは超頻出問題です。

メトリクスの基本概念

  • 名前空間（Namespace）: メトリクスのグループ分け。例: AWS/EC2、AWS/RDS、Custom/TsuriSpot
  • ディメンション（Dimension）: メトリクスを細分化する属性。例: InstanceId=i-12345 で特定のEC2を指定
  • 統計（Statistics）: Average（平均）、Sum（合計）、Maximum、Minimum、SampleCount、pNN（パーセンタイル）
  • 期間（Period）: データの集計間隔。標準5分、詳細モニタリング（有料）で1分

13.1.2 CloudWatch Alarms

メトリクスが閾値（しきいち）を超えたときに通知やアクションを実行します。

アラームの3つの状態:

  • OK: 閾値の範囲内（正常）
  • ALARM: 閾値を超えた（異常）
  • INSUFFICIENT_DATA: データ不足で判定不可

発動時のアクション:

  • SNS通知: メール・SMSで関係者に通知
  • Auto Scalingアクション: インスタンスを増減
  • EC2アクション: 停止・終了・再起動・復旧

コンポジットアラーム

複数のアラームをAND/OR条件で組み合わせた上位アラーム。「CPU 80%超え AND メモリ 70%超え」のときだけ通知、のように設定でき、アラームの嵐を防げます。

ツリスポのApp Runner監視: App RunnerのCPU使用率が80%を超えたらCloudWatch Alarm → SNS → 運営者メールに通知。コンポジットアラームで「CPU高 AND メモリ高」のときだけ緊急通知にすれば、一時的なスパイクによる誤報を減らせます。

13.1.3 CloudWatch Logs

アプリケーションやAWSサービスのログを集中管理するサービスです。

  • ロググループ（Log Group）: ログの大きなまとまり。アプリごとに1つ（例: /app-runner/tsurispot）
  • ログストリーム（Log Stream）: ロググループ内の個別ログの流れ。インスタンスごとに1つ
  • 保持期間: 1日〜10年、または無期限

主要機能

  • メトリクスフィルター: ログから特定パターン（"ERROR"等）を検出 → メトリクス化 → アラーム設定可能
  • Logs Insights: SQLライクなクエリでログ分析
  • S3エクスポート: 長期保管用（最大12時間の遅延あり）
  • サブスクリプションフィルター: リアルタイムで他サービス（Lambda、Kinesis Data Firehose、OpenSearch）に転送

S3エクスポート vs サブスクリプションフィルター: S3エクスポートは最大12時間遅延。リアルタイム処理にはサブスクリプションフィルター → Kinesis Data Firehose → S3 を使う。「リアルタイム」が出たらサブスクリプションフィルター！

13.1.4 CloudWatch Agent

EC2やオンプレミスサーバーにインストールし、OS内部の情報を取得してCloudWatchに送信するエージェントです。

  • カスタムメトリクス: メモリ使用率、ディスク使用率、スワップ使用率
  • ログ収集: アプリケーションログ、OSログをCloudWatch Logsに転送

定番問題:「EC2のメモリ使用率を監視したい」→ CloudWatch Agent。「アプリログをCloudWatch Logsに送りたい」→ CloudWatch Agent。エージェントが必要な場面は必ず出題されます。

13.1.5 その他のCloudWatch機能

  • Dashboards: メトリクス一覧表示。リージョン横断で表示可能
  • Contributor Insights: ログからTop-Nの要因を特定（最もエラーを出すIPなど）
  • Metric Streams: メトリクスをKinesis Data Firehose経由で外部（Datadog等）に配信

13.2 CloudTrail 〜誰が何をしたかの記録〜

AWS CloudTrailは、AWSアカウント内で「誰が」「いつ」「どのAPI操作を行ったか」を記録する監査ログサービスです。CloudWatchが「システムの状態」を監視するのに対し、CloudTrailは「人やプログラムの操作」を記録します。

CloudTrailは「監視カメラの録画」です。オフィスの出入り口のカメラが誰がいつ入退室したかを記録するように、AWSの全API操作を記録します。何か問題が起きたとき「誰がやったのか」を特定するために不可欠です。

イベントの種類

イベント種類内容デフォルト例
管理イベントリソースの作成・変更・削除有効（無料）EC2起動、IAMロール作成
データイベントリソース内のデータ操作無効（有料）S3のGet/Put、Lambda実行
インサイトイベント異常なAPIアクティビティ検出無効（有料）大量のTerminateInstances

CloudTrailのログ配信先: S3バケットに保存。CloudWatch Logsにも配信可能で、「ルートユーザーがログインしたら通知」のようなセキュリティ監視に使えます。ログの整合性チェック機能で改ざん検知もできます。

ツリスポのCloudTrail活用: 「誰がS3バケットの権限を変更したか」「いつApp Runnerが更新されたか」が全て記録されます。不正アクセス調査や、昨日のデプロイで何が変わったかの振り返りにも使えます。

13.3 EventBridge 〜イベント駆動の心臓部〜

Amazon EventBridge（旧CloudWatch Events）は、イベントを検知しターゲットにルーティングする「イベントバス」です。

EventBridgeは「郵便局の仕分けシステム」です。差出人（AWSサービスやアプリ）からの手紙（イベント）を、宛先ルールに基づいて届け先（Lambda、SQSなど）に仕分けます。

イベントバスの種類

  • デフォルトイベントバス: AWSサービスからのイベント
  • パートナーイベントバス: SaaSパートナーからのイベント
  • カスタムイベントバス: 自作アプリからのイベント

ルールとターゲット

  • イベントパターン: 特定イベントに反応（EC2停止、S3オブジェクト作成など）
  • スケジュール: cron式（cron(0 9 * * ? *) = 毎日UTC 9:00）やrate式で定期実行

ターゲット: Lambda、SQS、SNS、Step Functions、ECSタスク等150以上のサービス

EventBridgeのスキーマレジストリ: イベントの構造（スキーマ）を自動検出・保存する機能。開発者がイベント形式を調べる手間を省けます。

ツリスポの自動化: EventBridgeで毎週月曜朝9時にLambda起動 → microCMS APIで記事の公開状態を自動チェック。App Runnerデプロイ完了イベント → SNS → Slack通知。手作業ゼロの運用自動化が実現できます。

13.4 X-Ray 〜分散トレーシング〜

AWS X-Rayは、マイクロサービスでリクエストが複数サービスを通過する様子を可視化する「分散トレーシング」サービスです。API Gateway → Lambda → DynamoDB のどこで遅延が発生しているかを特定できます。

X-Rayの出題キーワード:「レイテンシの原因特定」「分散アプリのデバッグ」「サービスマップ」「トレース」「ボトルネック」→ 全てX-Ray。

13.5 Config / Trusted Advisor / Health Dashboard

AWS Config

リソースの設定変更を記録し、コンプライアンスルールへの準拠を評価。「S3がパブリックでないか」「EC2にタグがあるか」を自動チェックします。

CloudTrail vs Config: CloudTrail=「誰が何をした」（API操作ログ）、Config=「設定がどう変わった、正しいか」（設定変更履歴+評価）。

Trusted Advisor

5カテゴリでベストプラクティスを提案: コスト最適化/パフォーマンス/セキュリティ/耐障害性/サービス制限。

全チェック項目はBusiness以上のサポートプランが必要。Basic/Developerでは一部のみ。

Health Dashboard

  • Service Health: AWSサービス全体の稼働状況
  • Your Account Health: 自分のアカウントに影響するイベント通知

第13章まとめ

  • CloudWatch: メトリクス + ログ + アラームの統合監視
  • EC2のメモリ使用率はCloudWatch Agent必須（標準メトリクスに含まれない）
  • ログのリアルタイム転送はサブスクリプションフィルター。S3エクスポートは最大12h遅延
  • CloudTrail = API操作ログ（誰が何をしたか）
  • EventBridge = イベント駆動 + cron スケジュール
  • X-Ray = 分散トレーシング（ボトルネック特定）
  • Config = リソース設定の記録・評価（CloudTrailとは別物）
  • Trusted Advisor = 5カテゴリのBPチェック（Business以上で全機能）

Q1. EC2のメモリ使用率を監視するのに必要なものは？

答えを見る

CloudWatch Agentをインストールしカスタムメトリクスとして送信。メモリはOS内部情報のため標準メトリクスでは取得不可。

Q2. CloudWatch Logsをリアルタイムで処理するには？

答えを見る

サブスクリプションフィルター → Kinesis Data Firehose → S3/Lambda/OpenSearch。S3エクスポートは最大12h遅延で不適切。

Q3. 「誰がS3をパブリックにしたか」を調べるサービスは？

答えを見る

CloudTrail。API操作ログからPutBucketPolicy等の呼び出し元を特定できる。

Q4. CloudWatch / CloudTrail / Config の違いを一言ずつで。

答えを見る

CloudWatch=「今の状態は？」、CloudTrail=「誰が何をした？」、Config=「設定は正しい？」

14 メッセージングとデカップリング 〜マイクロサービス設計の基本〜

大規模なシステムでは、すべてのコンポーネント（部品）が直接つながっていると、一箇所の障害が全体に波及します。これを「密結合（きつけつごう）」と呼びます。AWSでは「メッセージング」サービスを使ってコンポーネント間を「疎結合（そけつごう）」にし、障害に強いシステムを作ります。

密結合: 電話のように相手がいないと通信できない。片方が忙しいと止まる。
疎結合: メールのように相手が不在でも送れる。相手は好きなタイミングで読む。
SQS/SNSはこの「メール」のような役割を果たし、送り手と受け手を切り離します。

14.1 SQS（Simple Queue Service）〜メッセージキュー〜

SQSは「キュー（待ち行列）」にメッセージを入れて、別のサービスが取り出すというシンプルな仕組みです。送り手（Producer）がメッセージをキューに入れ、受け手（Consumer）がキューからメッセージを取り出して処理します。

Standard Queue vs FIFO Queue

特性Standard QueueFIFO Queue
スループットほぼ無制限300 msg/秒（バッチで3,000）
順序保証ベストエフォート（順番が入れ替わることがある）厳密に順序保証
重複配信まれに重複配信あり（At-Least-Once）重複なし（Exactly-Once）
キュー名自由末尾に .fifo が必須
ユースケース大量処理、順序不問注文処理、金融取引など順序が重要な場面

SQSの重要パラメータ:

  • 可視性タイムアウト（Visibility Timeout）: メッセージ取得後、他のConsumerから見えなくなる時間。デフォルト30秒、最大12時間。処理時間より長く設定する
  • メッセージ保持期間: 1分〜14日（デフォルト4日）
  • 最大メッセージサイズ: 256KB。大きいデータはS3に保存しポインタ（URL）をSQSに送る
  • ロングポーリング: キューが空のとき最大20秒待ってからレスポンスを返す。API呼び出し回数とコストを削減。ショートポーリングは即座に空レスポンスを返す（無駄なAPI呼び出しが増える）

デッドレターキュー（DLQ）

処理に何度も失敗したメッセージを退避させる専用キューです。例えば「5回処理に失敗したメッセージはDLQに移動」と設定でき、問題のあるメッセージを後から調査できます。通常のキューの処理が詰まることを防ぐ安全弁です。

遅延キュー（Delay Queue）

メッセージを送信してから一定時間（0秒〜15分）はConsumerに見えなくする設定。「注文後5分間はキャンセル可能にしたい」といった場面で使います。

SQS + Auto Scaling

SQSのキュー内メッセージ数（ApproximateNumberOfMessages）をCloudWatchメトリクスとして監視し、メッセージが増えたらEC2やECSタスクをスケールアウトする構成です。

ツリスポの釣果投稿処理: 将来、ユーザーが釣果写真を投稿できる機能を追加した場合の構成例。ユーザー → API Gateway → SQS（Standard Queue）→ Lambda（画像リサイズ+DynamoDB保存）。大量投稿が来てもSQSがバッファになるので、Lambdaが処理しきれなくても投稿は受け付けられます。5回リサイズに失敗した画像はDLQに退避して後で調査。

14.2 SNS（Simple Notification Service）〜Pub/Subメッセージング〜

SNSは「Pub/Sub（パブリッシュ/サブスクライブ）」モデルのメッセージングサービスです。Publisher（発行者）がトピック（テーマ）にメッセージを発行し、そのトピックを購読しているSubscriber（購読者）全員にメッセージが届きます。1対多の配信です。

サブスクリプションの種類

  • SQS: キューにメッセージを送信
  • Lambda: 関数を呼び出し
  • HTTP/HTTPS: WebhookでURLに通知
  • Email / Email-JSON: メール送信
  • SMS: ショートメッセージ送信
  • Kinesis Data Firehose: ストリームに配信

SNS + SQS ファンアウトパターン

1つのSNSトピックに複数のSQSキューをサブスクライブさせ、1回のメッセージ送信で複数の処理を並行実行するパターンです。これは試験で超頻出のアーキテクチャパターンです。

ファンアウトパターン: 「1つのイベントを複数のサービスに同時に配信したい」→ SNS + SQS ファンアウト。SNSトピックに複数のSQSキューをサブスクライブし、各キューから独立した処理を実行。これにより処理間の依存がなくなり、片方が失敗しても他方に影響しません。

メッセージフィルタリング

サブスクリプションにフィルターポリシーを設定し、特定の属性を持つメッセージだけを受信できます。例: 注文トピックで「東京」の注文だけを受け取るSQSキュー。

ツリスポのファンアウト: ユーザーが釣果を投稿したら、SNSトピック「釣果投稿」に1回発行。サブスクライバーとして (1) SQS→DynamoDB保存、(2) SQS→画像リサイズ+S3保存、(3) SQS→フォロワーへのプッシュ通知、(4) Lambda→アクセスログ記録。各処理は完全に独立しているので、画像リサイズが失敗しても通知やDB保存には影響しません。

14.3 Kinesis 〜リアルタイムストリーミング〜

Amazon Kinesisは、大量のストリーミングデータ（リアルタイムに流れ続けるデータ）を収集・処理するサービス群です。

Kinesisファミリー

サービス用途特徴
Kinesis Data Streamsリアルタイム処理シャード単位のスケーリング。複数Consumerが同じデータを読める。24h〜365日保持
Kinesis Data Firehose配信（S3/Redshift/OpenSearch等）ニアリアルタイム（60秒バッファ）。完全マネージド、自動スケーリング
Kinesis Data Analyticsストリーム分析SQLまたはApache Flinkでリアルタイム分析

SQS vs Kinesis の使い分け（超頻出！）

観点SQSKinesis Data Streams
処理モデルメッセージを取り出して消す（キュー）データを読むが消えない（ストリーム）
Consumer数1つのメッセージは1Consumerが処理複数Consumerが同じデータを読める
順序Standard: 順序保証なし / FIFO: 保証ありシャード内で順序保証
スケーリング自動シャード数を手動で管理
データ保持最大14日24h〜365日
ユースケースタスクキュー、非同期処理ログ集約、IoT、リアルタイム分析

キーワード: 「デカップリング」「非同期処理」→ SQS。「リアルタイム」「ストリーミング」「複数Consumer」→ Kinesis。

14.4 Amazon MQ 〜レガシー移行向けメッセージブローカー〜

Amazon MQは、Apache ActiveMQやRabbitMQのマネージドサービスです。オンプレミスで既にActiveMQ/RabbitMQを使っている場合の移行先として使います。新規開発ではSQS/SNSを使うのがAWSのベストプラクティスです。

Amazon MQの出題パターン:「オンプレミスのActiveMQ/RabbitMQをAWSに移行」「MQTT/AMQP/STOMPプロトコルを使いたい」→ Amazon MQ。新規開発でメッセージングが必要ならSQS/SNS。

第14章まとめ

  • SQS: メッセージキュー。Standard（高スループット）vs FIFO（順序保証）
  • 可視性タイムアウト、DLQ、遅延キュー、ロングポーリングは試験頻出
  • SNS: Pub/Subメッセージング。1対多の配信
  • ファンアウトパターン: SNS → 複数SQS で1イベントを複数処理に分散
  • Kinesis: リアルタイムストリーミング。SQSとの使い分けを確実に
  • SQS vs Kinesis:「非同期処理」→SQS、「リアルタイム+複数Consumer」→Kinesis
  • Amazon MQ: ActiveMQ/RabbitMQの移行用。新規はSQS/SNS
  • メッセージサイズ上限はSQS=256KB。大きいデータはS3+ポインタ

Q1. SQSのStandard QueueとFIFO Queueの最大の違いは？

答えを見る

Standard=ほぼ無制限スループットだが順序保証なし・重複あり。FIFO=厳密な順序保証・重複なしだが300msg/秒の制限あり。

Q2. 1つのイベントを複数のサービスに同時配信するアーキテクチャパターンは？

答えを見る

SNS + SQS ファンアウトパターン。SNSトピックに複数のSQSキューをサブスクライブさせる。

Q3. SQSとKinesisの使い分けの基準は？

答えを見る

SQS=「非同期処理」「デカップリング」「タスクキュー」。Kinesis=「リアルタイムストリーミング」「複数Consumer」「ログ集約」「IoT」。

Q4. SQSで処理に5回失敗したメッセージを退避させる仕組みは？

答えを見る

デッドレターキュー（DLQ）。maxReceiveCount=5に設定すると、5回処理に失敗したメッセージは自動的にDLQに移動する。

Q5. オンプレミスのRabbitMQをAWSに移行する場合のサービスは？

答えを見る

Amazon MQ。既存のActiveMQ/RabbitMQのマネージドサービスで、プロトコル互換性がある。新規開発ならSQS/SNS。

15 データ分析サービス 〜ビッグデータを活用する〜

AWSには膨大なデータを分析するためのサービスが多数あります。SAA試験では「どのサービスをどの場面で使うか」の選択が問われます。ここでは各サービスの特徴と使い分けを明確にしましょう。

データ分析を「釣りの記録」に例えると: 生データ（釣果記録ノート）→ ETL（ノートを整理してExcelに入力）→ データレイク（Excelファイルの保管庫）→ 分析（ピボットテーブルで集計）→ 可視化（グラフ作成）。AWSにはこの各段階に特化したサービスがあります。

15.1 Athena 〜S3にSQLを実行〜

Amazon Athenaは、S3に保存されたデータに対して直接SQLクエリを実行できるサーバーレスサービスです。サーバーの構築やデータベースへのロードが不要で、S3にデータを置くだけですぐにクエリできます。

Athenaの特徴

  • サーバーレス: インフラ管理不要
  • 課金体系: スキャンしたデータ量に対して課金（$5/TB）
  • 対応形式: CSV、JSON、Parquet、ORC、Avroなど
  • Glueカタログ連携: テーブル定義をGlue Data Catalogで管理

Athenaのコスト最適化（超頻出）: データをParquetやORCなどの列指向フォーマットに変換すると、必要な列だけスキャンするためコストが劇的に下がります。CSVの100分の1以下になることも。さらにデータをパーティション分割（例: 年/月/日でフォルダ分け）すると、スキャン範囲が限定されてさらにコスト削減。試験では「Athenaのコスト最適化」=「Parquet/ORC変換 + パーティション」が正解です。

ツリスポのアクセスログ分析: CloudFrontのアクセスログをS3に保存 → Athenaで「どのスポットページが最もアクセスされているか」「どの地域からのアクセスが多いか」をSQLで集計。Parquet形式にしておけば月間50万PVのログ分析でもクエリ1回あたり$0.01以下で済みます。

15.2 AWS Glue 〜ETLとデータカタログ〜

AWS Glueは、データの抽出・変換・ロード（ETL: Extract, Transform, Load）を行うサーバーレスサービスです。

主要コンポーネント

  • Glue Data Catalog: データのメタデータ（テーブル定義、スキーマ情報）を管理する中央カタログ。Athena、Redshift Spectrum、EMRから参照される
  • Glue Crawler: S3やRDSなどのデータソースを自動スキャンし、スキーマを検出してData Catalogに登録する
  • Glue Job: PythonやSparkでデータ変換処理を実行（例: CSVをParquetに変換）
  • Glue Studio: ビジュアルなGUIでETLジョブを作成

Glue Data CatalogはAthenaに必須: AthenaでS3データにクエリするには、まずGlue Data Catalogでテーブル定義を作成する必要があります。Crawlerで自動検出するか、手動でテーブルを定義します。

15.3 EMR（Elastic MapReduce）

Amazon EMRは、Apache Hadoop、Apache Spark、Prestoなどのビッグデータフレームワークをマネージドで実行するサービスです。ペタバイト級のデータ処理に使います。

Athena vs EMR: 軽量な分析（アドホッククエリ）→ Athena。大規模で複雑な処理（機械学習、大量のETL）→ EMR。EMRはクラスターの管理が必要なため、手軽さではAthenaに劣ります。

15.4 Redshift 〜データウェアハウス〜

Amazon Redshiftは、大規模データの分析に特化した列指向のデータウェアハウスサービスです。ペタバイト級のデータに対して高速なSQLクエリを実行できます。

Redshiftの特徴

  • 列指向ストレージ: 分析クエリ（特定の列を集計するなど）に最適化
  • ノードベースの課金: 常時稼働型（Athenaのようなクエリ単位ではない）
  • Redshift Spectrum: Redshiftから直接S3のデータにクエリ（Redshiftにロード不要）
  • Redshift Serverless: サーバーレス版。使った分だけ課金

Athena vs Redshift の使い分け:

  • Athena: アドホック（その場限り）クエリ、不定期な分析、小〜中規模データ、サーバーレス
  • Redshift: 定期的な大量データ分析、複雑な結合、ダッシュボード用の高速クエリ、常時稼働

キーワード: 「サーバーレス」「S3上のデータに直接クエリ」「アドホック」→ Athena。「データウェアハウス」「ペタバイト」「BI」「高速ダッシュボード」→ Redshift。

15.5 OpenSearch（旧Elasticsearch）

Amazon OpenSearch Serviceは、全文検索とログ分析のためのマネージドサービスです。Kibanaダッシュボード（OpenSearch Dashboards）で可視化もできます。

よくあるパターン: CloudWatch Logs → サブスクリプションフィルター → Lambda → OpenSearch → ダッシュボードで可視化

OpenSearchの出題キーワード:「全文検索」「ログ分析とダッシュボード」「Elasticsearch」「Kibana」→ OpenSearch。Athenaとの違い: AthenaはS3上のデータへのSQLクエリ、OpenSearchは全文検索とリアルタイムログ分析に特化。

15.6 QuickSight 〜BIダッシュボード〜

Amazon QuickSightは、BI（ビジネスインテリジェンス）ダッシュボードサービスです。様々なデータソース（Athena、Redshift、RDS、S3など）からデータを取得し、グラフやチャートで可視化します。機械学習ベースのインサイト機能（ML Insights）もあります。

15.7 その他のデータ分析サービス

  • Lake Formation: データレイク（S3上の大規模データ保管庫）の構築・管理・セキュリティを簡素化するサービス
  • MSK（Managed Streaming for Apache Kafka）: Apache Kafkaのマネージドサービス。Kinesisの代替で、既にKafkaを使っている環境の移行先
  • Data Pipeline: データの移動・変換をスケジュール実行するオーケストレーションサービス（レガシー。新規はStep Functions + Glueが推奨）

ツリスポのデータ分析パイプライン: CloudFrontアクセスログ → S3に蓄積 → Glue CrawlerでスキーマをData Catalogに登録 → Glue JobでParquet形式に変換 → AthenaでSQL集計（人気スポットランキング、地域別アクセス数）→ QuickSightでダッシュボード化。月間50万PVのログをParquet形式でクエリすれば1回あたり$0.01以下。年間コストは数ドルで済みます。

第15章まとめ

  • Athena: S3にSQL。サーバーレス、スキャン量課金。Parquet/ORC変換でコスト大幅削減
  • Glue: ETL + Data Catalog + Crawler。AthenaやRedshiftのメタデータ管理
  • EMR: Hadoop/Sparkの大規模ビッグデータ処理
  • Redshift: 列指向DWH。大規模・定期分析・高速BI向け
  • OpenSearch: 全文検索 + リアルタイムログ分析 + ダッシュボード
  • QuickSight: BIダッシュボード。多数のデータソースに対応
  • Athena vs Redshift: 「アドホック・サーバーレス」→ Athena、「大規模・常時」→ Redshift

Q1. AthenaのクエリコストをCSV対比で大幅に下げる方法は？

答えを見る

データをParquetまたはORC（列指向フォーマット）に変換する。必要な列だけスキャンするため、コストが数十〜数百分の1になる。パーティション分割も有効。

Q2. S3のデータのスキーマを自動検出してAthenaで使えるようにするには？

答えを見る

Glue Crawlerを実行してスキーマを自動検出し、Glue Data Catalogにテーブル定義を登録する。AthenaはGlue Data Catalogを参照してクエリを実行する。

Q3. アドホッククエリにAthena、常時稼働の大規模分析にRedshiftを使う理由は？

答えを見る

Athenaはスキャン量課金のサーバーレスで不定期利用に最適。Redshiftはノード課金で常時稼働だが、大規模データの複雑なクエリを高速処理できる。

16 移行と転送 〜オンプレミスからクラウドへ〜

多くの企業がオンプレミス（自社サーバー）からAWSへの移行を検討しています。SAA試験では「どの移行戦略を選ぶか」「どのデータ転送サービスを使うか」が頻出です。

16.1 クラウド移行の6R戦略

AWSが提唱する移行戦略は6つあり、「6R」と呼ばれます。アプリケーションごとに最適な戦略を選びます。

戦略内容変更度例
Rehost
(リフト&シフト)		そのままAWSに移行最小EC2にそのまま載せ替え
Replatform
(リフト&最適化)		少し最適化して移行小DBをRDSに変更、OSはそのまま
RepurchaseSaaS製品に置き換え中自社CRM → Salesforce
Refactor
(リアーキテクト)		クラウドネイティブに再設計大モノリス → マイクロサービス
Retire廃止-使われていないシステムを停止
Retain現状維持（移行しない）-移行が困難なレガシーシステム

ツリスポのVercel→AWS移行: Replatform戦略を採用しました。コード自体は大きく変更せず（Next.js standalone）、インフラだけVercel → App Runner（Docker）に変更。データベースを後から導入する場合は、オンプレMySQL → Aurora PostgreSQL への異種DB移行にDMSを使うことになります。

16.2 DMS（Database Migration Service）

AWS DMSは、データベースの移行を行うサービスです。ソース（移行元）からターゲット（移行先）にデータを転送します。

DMSの特徴

  • 同種DB間移行: MySQL → MySQL、Oracle → Oracle（簡単）
  • 異種DB間移行: Oracle → Aurora PostgreSQL（SCTが必要）
  • CDC（Change Data Capture）: 初期データロード後も、差分を継続的にレプリケーション（ダウンタイム最小化）
  • ソースDBは稼働したまま移行可能

SCT（Schema Conversion Tool）

異種DB間移行で必要なスキーマ変換ツール。OracleのストアドプロシージャをPostgreSQL形式に変換するなど、スキーマレベルの変換を自動化します。

DMS出題パターン:「ダウンタイムを最小限にDBを移行」→ DMS + CDC。「Oracle → Aurora PostgreSQL」→ DMS + SCT。「移行中もソースDBは稼働」→ DMSの特徴。

16.3 Snow Family 〜物理デバイスでのデータ移行〜

ネットワーク経由では時間がかかりすぎる大量データの移行に、AWSが物理デバイスを郵送して使う「Snow Family」があります。

デバイス容量用途
Snowcone8TB HDD / 14TB SSD小規模移行、エッジコンピューティング。2.1kgで最小・最軽量
Snowball Edge Storage Optimized80TB中規模データ移行。EC2互換のコンピューティング機能あり
Snowball Edge Compute Optimized42TBエッジでの処理が必要な場合。より強力なCPU/GPU
Snowmobile100PB超大規模移行。トラック1台で運搬。10PB以上ならSnowmobile推奨

Snow Familyの選択基準: 数TB → Snowcone。数十TB〜数百TB → Snowball Edge。10PB以上 → Snowmobile。「ネットワーク帯域が限られている」「数週間以上かかる」→ Snow Family。転送にかかる日数の計算問題も出ます。

16.4 DataSync 〜高速データ転送〜

AWS DataSyncは、オンプレミスとAWS間、またはAWSサービス間のデータ転送を高速に行うサービスです。NFS、SMB、HDFS、S3、EFS、FSxに対応しています。

DataSync vs Snow Family: DataSyncはネットワーク経由（オンライン転送）。帯域が十分あれば高速。Snow Familyは物理デバイス（オフライン転送）。ネットワークが遅い・大量データのときに使う。

16.5 その他の移行サービス

  • Transfer Family: SFTP/FTPS/FTPでS3やEFSにファイル転送。既存のファイル転送ワークフローをそのまま使える
  • Migration Hub: 移行プロジェクト全体の進捗を一元管理するダッシュボード
  • Application Discovery Service: オンプレミス環境を調査し、サーバーの依存関係やリソース使用状況をマッピング
  • Application Migration Service（MGN）: サーバーのRehost（リフト&シフト）移行を自動化
  • Outposts: AWSのインフラ（ラック）をオンプレミスに物理的に設置。低レイテンシやデータ主権要件に対応

移行サービスの使い分け:

  • 「サーバーをそのまま移行（Rehost）」→ MGN
  • 「DB移行」→ DMS
  • 「大量データ（TB/PB級）をオフラインで」→ Snow Family
  • 「NFS/SMBデータをオンラインで」→ DataSync
  • 「FTP/SFTPで既存ワークフローのまま」→ Transfer Family
  • 「オンプレ環境を調査」→ Application Discovery Service
  • 「AWSをオンプレに設置」→ Outposts

ツリスポの移行を例に考える: もしツリスポがオンプレのMySQLに2,000件のスポットデータを持っていた場合、Aurora PostgreSQLへの移行はDMS+SCTで実現します。画像データが数百GBあればDataSyncでS3に転送。数PBの動画データがあればSnowball Edgeを使います。移行全体の進捗はMigration Hubで管理します。

第16章まとめ

  • 6R戦略: Rehost/Replatform/Repurchase/Refactor/Retire/Retain
  • DMS: DB移行。CDC（継続レプリケーション）でダウンタイム最小化。異種DBはSCT併用
  • Snow Family: 物理デバイスで大量データ移行。Snowcone(〜14TB)/Snowball(〜80TB)/Snowmobile(100PB)
  • DataSync: オンラインデータ転送（NFS/SMB/S3/EFS/FSx対応）
  • Transfer Family: SFTP/FTPでS3/EFSへ転送
  • MGN: サーバーのRehost移行を自動化
  • Outposts: AWSインフラをオンプレミスに設置

Q1. Oracle → Aurora PostgreSQL の移行に必要なサービスの組み合わせは？

答えを見る

DMS（データ転送）+ SCT（スキーマ変換）。異種DB間移行ではSCTでスキーマを変換してからDMSでデータを転送する。

Q2. 50TBのデータをAWSに移行。ネットワーク帯域が100Mbpsで転送に46日かかる場合の最適な方法は？

答えを見る

Snowball Edge Storage Optimized（80TB）を使用。物理デバイスの配送・データ転送・返送で1〜2週間程度と、ネットワーク転送の46日より大幅に短縮。

Q3. オンプレミスのNFSサーバーからS3へ定期的にデータを同期する場合は？

答えを見る

AWS DataSync。NFS/SMBに対応し、増分転送、スケジュール実行、データ整合性チェックが可能。

17 コスト最適化とWell-Architected Framework 〜賢くAWSを使う〜

SAA試験の約20%はコスト最適化に関する問題です。また、Well-Architected Frameworkは試験全体を通じて問われる「設計思想」の基盤です。この章は試験の得点源になりますので、しっかり理解しましょう。

17.1 Well-Architected Framework の6本柱

AWSが推奨する「良い設計」の指針で、6つの柱（ピラー）で構成されています。

柱概要関連サービス例
1. 運用上の優秀性
(Operational Excellence)		運用の自動化、改善の継続CloudFormation, Systems Manager, CodeDeploy, Config
2. セキュリティデータ保護、権限管理、検知IAM, KMS, VPC, Shield, WAF, CloudTrail, GuardDuty
3. 信頼性
(Reliability)		障害からの自動復旧、スケーリングMulti-AZ, Auto Scaling, Route 53, バックアップ
4. パフォーマンス効率リソースの効率的な使用Auto Scaling, CloudFront, ElastiCache, Lambda
5. コスト最適化不要な支出の排除RI, Savings Plans, Spot, S3ライフサイクル, Compute Optimizer
6. サステナビリティ
(2022年追加)		環境への影響の最小化適切なリージョン選択、効率的なリソース使用

Well-Architected Tool: ワークロード（アプリケーション）を6本柱に基づいてレビューし、改善点を提案するAWSのツール。試験では「ワークロードのベストプラクティスへの準拠を確認したい」→ Well-Architected Toolが正解。

17.2 コスト管理サービス

AWS Cost Explorer

過去のコストデータを分析し、将来のコストを予測するツール。サービス別、リージョン別、タグ別などで可視化できます。

AWS Budgets

予算を設定し、予算の超過（またはその予測）をSNSやメールで通知。実コスト予算、使用量予算、RI/Savings Plansの活用率予算を設定可能。

Cost and Usage Report（CUR）

最も詳細なコストデータ。1時間単位のリソースレベルまでの明細が含まれ、S3に出力されます。Athenaで分析可能。

AWS Compute Optimizer

EC2、EBS、Lambda、ECSのリソースサイズが適切かどうかを機械学習で分析し、最適なサイズを推奨します。

17.3 EC2購入オプション（コスト最適化の核心）

購入オプション割引率期間特徴
オンデマンド0%（定価）なし従量課金。いつでも開始・停止可能
Reserved Instances (RI)最大72% OFF1年 or 3年インスタンスタイプ・リージョン・OSを予約。全額前払い/一部前払い/前払いなし
Savings Plans最大72% OFF1年 or 3年コンピューティング使用量（$/時）を約束。EC2/Fargate/Lambdaに適用。RIより柔軟
Spot Instances最大90% OFFなし余剰キャパシティ。2分前通知で中断される可能性あり。バッチ処理、CI/CD、ビッグデータ分析など中断耐性のあるワークロード向け
Dedicated Hosts--物理サーバーを専有。ライセンス要件（BYOL）やコンプライアンス要件
Dedicated Instances--ハードウェアは専有だが物理サーバーの指定はできない

RI vs Savings Plans: どちらも1年/3年のコミットで最大72%割引。RIはインスタンスタイプが固定、Savings Plansは使用量（$/h）を約束するので柔軟。試験では「最もコスト効率が良い」=「3年全額前払いのRI or Savings Plans」、「柔軟性が必要」=「Savings Plans」が正解パターン。

Spot Instancesの注意点: 最大90%割引だが中断される可能性がある。「中断耐性」が必要。本番Webサーバーには不適切。バッチ処理、データ分析、CI/CD、テスト環境に使う。Spot Fleet（複数Spotの組み合わせ）やEC2 Auto Scaling Groupのミックスインスタンスポリシーで活用。

ツリスポのコスト最適化: App Runnerは使った分だけ課金（アイドル時は最小インスタンスのみ）で、小規模サイトには最適。将来トラフィックが増えてEC2に移行するなら、Savings Plans（3年、全額前払い=最大72%OFF）を検討。SSGビルドのようなバッチ処理にはSpot Instancesを使えば最大90%OFF。S3の画像はライフサイクルポリシーで30日後にIA、90日後にGlacierへ自動移行。

17.4 インフラ自動化（IaC）

CloudFormation

Infrastructure as Code（IaC）サービス。YAMLまたはJSON形式のテンプレートでAWSリソースを定義し、「スタック」として一括作成・更新・削除できます。

  • テンプレート: リソース定義（必須セクション: Resources）
  • スタック: テンプレートから作成されたリソースの集合
  • スタックセット: 複数アカウント・リージョンに同じスタックを展開
  • ドリフト検出: テンプレートと実際のリソースの差異を検出

CloudFormationの出題パターン:「インフラの再現性」「複数環境を同じ構成で」「設定をバージョン管理」→ CloudFormation。テンプレートのResourcesセクションは唯一の必須セクション。

CDK（Cloud Development Kit）

TypeScript、Python、Javaなどのプログラミング言語でインフラを定義し、CloudFormationテンプレートを生成するツール。プログラマにとってはYAMLより書きやすい。

Elastic Beanstalk

PaaS（Platform as a Service）。コードをアップロードするだけで、EC2、ALB、Auto Scaling、RDSなどのインフラを自動構築。開発者はコードに集中できます。裏ではCloudFormationが動いています。

SAM（Serverless Application Model）

サーバーレスアプリ（Lambda + API Gateway + DynamoDB等）に特化したIaCフレームワーク。CloudFormationの拡張版です。

Systems Manager

EC2やオンプレミスサーバーの運用を自動化するサービス群。

  • Run Command: コマンドをリモート実行（SSHなしで）
  • Patch Manager: OSパッチの自動適用
  • Session Manager: ブラウザからシェルアクセス（SSHポート不要、セキュア）
  • Parameter Store: 設定値やシークレットの安全な保存
