# 04. 本番への反映（デプロイ）

> 「変更がどうやって本物のサイトに反映されるか」。仕組みだけ理解すればトラブル時に慌てません。戻し方も。

最終更新: 2026-07-07

---

## 🚀 デプロイとは
**デプロイ**＝作った変更を本物のサイト（tsurispot.com）に反映すること。ほぼ**自動**です。

```
① コードをGitHubの「master」に反映（マージ）
      ↓ 自動スタート
② GitHub Actions がビルド（組み立て）
      ↓
③ 完成品をAWSサーバー（App Runner）に送る
      ↓
④ サイトが新しい内容に切り替わる（10〜15分）
```

あなたがやるのは基本 **①のマージ（緑のボタンを押す）だけ**。

---

## 📝 変更を本番に出す手順
**重要ルール：`master` に直接変更を入れてはいけません**（過去に直接いじって障害多発）。

1. 作業用の枝（ブランチ）`feature/○○` を作る
2. 変更・コミット・push
3. **PR（プルリクエスト）** を作る＝「masterに入れていい？」の提案
4. 内容を確認して **「Merge」ボタンを押す**（あなたの承認）
5. 自動でデプロイ（10〜15分）
6. 完了後、本番で確認

> 💡 コードを書くのは開発者/Claude。**あなたの役割は「PRを見てMergeを押す」承認者**です。

### PRを見るときのポイント
- タイトルと説明で「何をする変更か」わかるか
- 「変更ファイル」の数が説明と釣り合っているか（説明は小さいのにファイル大量＝要注意）
- 迷ったら「これは何の変更？」と聞く

---

## ⚙️ 仕組みの中身
- **どこで動く**：GitHub Actions（リポジトリ上部「Actions」タブ）
- **成功/失敗**：緑✓＝成功、赤✗＝失敗
- **時間**：10〜15分
- **`.md`（文書）だけの変更ではデプロイは走らない**（このハンドブック更新では再デプロイされません）

### 環境変数（設定値）
広告の名札やAPIキーは、GitHubの「**Secrets**」に保管。ビルド時にサイトへ焼き込まれるので、**値を変えたら次のデプロイで反映**されます。

---

## ↩️ 壊れたときの戻し方（ロールバック）
### 方法A（推奨・安全）：取り消すPRを作る
「さっきの変更を取り消す」PRを作ってマージ（技術的には `git revert`）。開発者/Claudeに「直前のデプロイをrevertして」と頼めばOK。

### 方法B（緊急）：サイトが完全に止まっているとき
`hotfix/○○` で素早く修正PR。開発者に依頼。

> ⚠️ **自分でコマンドをいじって直そうとしない**。症状（何が・いつから）を伝える方が最短で安全。

---

## 🩺 デプロイが失敗したとき
1. GitHub → Actions → 赤✗の「Deploy to App Runner」
2. ログの赤い部分。よくある原因：型エラー／テスト失敗／ビルドが重すぎる
3. 原因不明ならログを開発者/Claudeへ

**大事**：デプロイが失敗しても**本番は古いまま動き続けます**。「失敗＝ダウン」ではありません。

---

## 用語ミニ辞典
- **デプロイ**：変更を本番に反映すること。
- **master**：本番に直結する正式版。直接いじらない。
- **ブランチ（枝）**：作業用のコピー。
- **PR（プルリクエスト）**：「masterに入れていい？」の提案・レビュー依頼。
- **マージ**：PRを承認してmasterに取り込むこと。
- **ロールバック/revert**：変更を取り消して元に戻すこと。
- **Secrets**：APIキーなど秘密の設定値の保管場所。
